### D1 – 修复策略计时器（`policy_time_us_avg` 为零）

**问题根源**：策略计时代码（`_edgekv_note_policy_time`）依赖环境变量 `EDGEKV_H1_PROFILE_POLICY_TIME` 为真值才生效，但当前所有运行入口（`run_h1_vllm0110_real.py` 和 `_runner.py` 调用的 `run_real_cell`）均未显式设置该变量，导致计时分支被跳过。

**修改思路**：
- 在 **`_runner.py` 的 `run_real_cell` 函数**中，构建子进程环境时强制添加 `EDGEKV_H1_PROFILE_POLICY_TIME=1`，确保所有通过该 helper 启动的实验都开启计时。
- 在 **`run_h1_vllm0110_real.py` 的 `run_cell` 函数**中，同样在设置 `os.environ` 的部分增加该变量，作为直接运行该脚本时的保护措施。
- 这样无需改动 `sitecustomize.py` 中的计时逻辑，只需保证变量存在。

### D3 – 坐实退化：输出 `c_recomp` 线性、驱逐粒度、`score`/`p_reuse` 直方图

**问题根源**：现有代码已包含 `validate_d3.py` 脚本，可以从 `edgekv_gpu_stats_*.json` 文件中提取所需信息并生成 JSON 报告，但在实验流程中未被调用；同时，`sitecustomize.py` 中 `get_edgekv_gpu_cache_stats` 默认不输出 `object_profiles`，导致 `validate_d3.py` 能读取的样本数可能不足。

**修改思路**：
1. **开启对象级 profile 记录**：在实验运行（`run_h1_vllm0110_real.py` 的 `run_cell`）中设置环境变量 `EDGEKV_H1_STATS_INCLUDE_OBJECT_PROFILES=1`，使每个 `edgekv_gpu_stats_*.json` 包含所有对象的 `p_reuse`、`score`、`n_tokens`、`c_recomp_ms` 等字段，供后续分析。
2. **在实验汇总阶段调用 `validate_d3.py`**：
   - 在 `run_step3_real.py` 和 `run_step3_repeat.py` 的结尾（生成 summary CSV 之后），使用 `R.summarize` 调用 `validate_d3.py`，传入聚合后的 stats 目录（通常是 `BASE/edgekv_gpu_stats` 或各 cell 的 stats 目录），并指定输出 JSON 路径，例如 `d3_validation.json`。
   - 如果存在多个 rep 的 stats 目录，可以选择合并所有 JSON 文件后再调用，或直接对所有文件进行聚合（`validate_d3.py` 已支持遍历目录下所有 `edgekv_gpu_stats_*.json`，因此传入顶层 stats 目录即可）。
3. **验证报告内容**：生成的 JSON 将包含：
   - `c_recomp_per_token_mean` 与 `std`，并给出 `c_recomp_linear_ok`（判断是否线性）。
   - `eviction_granularity_values` 和 `block_granularity_ok`（确认驱逐粒度为 block 级）。
   - `score_p_reuse_corr` 及 `score_p50`、`score_std`、`p_reuse_std`，并给出 `score_p_reuse_degenerate_ok`（判断 score 是否只随 p_reuse 变化）。
   - `score_histogram` 和 `p_reuse_histogram` 直方图数据。
