# LPE Object-Level Score / Admission / Eviction 修复计划

## Summary

第一版实现“统一 score 口径 + object-level 诊断准入 + object-level 成组驱逐”。不直接篡改 vLLM cache/hash 映射，避免破坏内部状态；低分对象仍可进入 prefix cache，但会被标记为逻辑 reject，并在压力下按 object score 成组优先回收。

第一版稳定后，再用独立开关升级到“强制拒绝”。

## Key Changes

### 1. 统一 LPE score 口径

- 在 `h1/sitecustomize.py` 中拆分 `logical_size_mb` 与 `resident_size_mb`。
- `score = p_reuse * c_recomp_ms / logical_size_mb`，不再用当前 resident block bytes 做分母。
- `resident_size_mb/resident_block_count` 只用于诊断和 object-level eviction 组大小，不参与 score 分母。
- stats 中保留/新增 `logical_size_mb`、`resident_size_mb`、`score_size_source`，用于验证 score 不再被残留 block 放大。

### 2. 实现 object-level 诊断准入

- 在 `cache_full_blocks(...)` 中，在调用/完成 vLLM 原生 cache 登记后，按 object profile 计算 `admission_decision`。
- 判据：新对象 score 大于当前 resident object 的最低 score 时记为 `accept`，否则记为 `reject`。
- `reject` 不阻止 vLLM 缓存，只给 profile 标记：
  - `admission_rejected=True`
  - `admission_seq`
  - `admission_min_resident_score`
- `admission_rejection_count` 统计逻辑拒绝数量。
- runtime monitor 记录 `admit_accept/admit_reject`。

### 3. 实现 object-level 成组驱逐

- 维护 object -> resident block ids/group ids 的索引，基于 object score 构建 object rank，而不是直接以单个 block score 作为 LPE victim。
- `_edgekv_reorder_free_queue(...)` 中 LPE 路径按 object score 选择低价值 victim object，再把该 object 的所有 currently-free/resident-candidate blocks 作为连续组排到 free queue 前端。
- pinned object 不作为 victim。
- 逻辑 rejected object 在同等 score 下优先驱逐。
- 保留 LFU/LRU 原行为；只改变 `h1_lpe` 路径。

### 4. 第一版后续升级点

- 增加 `H1_LPE_ADMISSION_MODE=diagnostic|strict`，默认 `diagnostic`。
- 第一版稳定后实现 `strict`：
  - 低分对象跳过或缩短 `original_cache_full_blocks(...)` 的缓存登记。
  - 作为单独实验开关，不影响默认行为。

## Test Plan

### 单元测试

- score 使用 `logical_size_mb`，当 resident blocks 从 2 降到 1 时 score 不变，`resident_size_mb` 变化。
- object-level reorder 会把同一 object 的多个 blocks 成组排到 free queue 前端。
- 低 score 新对象标记 `admission_rejected=True`，但仍完成 vLLM 原生 cache 登记路径。
- rejected object 在同分或近似同分时优先于 accepted object 被成组驱逐。
- pinned object 不被选为 object victim。

- 当前阶段不直接删除 vLLM 内部 cache/hash/block 映射；object-level eviction 通过 free queue 成组排序实现。

- `h1_lru`、`h1_lfu` 行为保持兼容；新增逻辑只在 `EDGEKV_H1_GPU_POLICY=h1_lpe` 生效。
- 强制拒绝作为第二阶段独立开关实现，需在第一版稳定后再做。


# 第一版修复后开销分析

## Summary

对比第一版修复后的 `h1/out/lpe_smoke/sharedgpt_v5/step3_summary.csv` 与未修复的 `h1/备份-运行结果/H1完整实验_batch_size_8/sharedgpt_v5/step3_summary.csv`，同 budget 下 `policy_time_us_avg` 与 TTFT 都有小幅增加：

| budget | policy_time_us_avg | mean_ttft_ms | p95_ttft_ms | hit_rate |
|---|---:|---:|---:|---:|
| 0.75 old -> new | 177.11 -> 192.95, +15.84us | 714.80 -> 718.83, +4.03ms | 935.97 -> 947.74, +11.78ms | 0.5537 -> 0.5553 |
| 0.825 old -> new | 172.51 -> 187.60, +15.10us | 421.19 -> 430.61, +9.42ms | 707.29 -> 717.56, +10.27ms | 0.7529 -> 0.7530 |

hit rate 没有下降，甚至略升；TTFT 增加主要同步体现在 `prefill_ms`，不是 queue wait 增加。

## Cause

第一版是 diagnostic admission：新增了对象级 profile、score、resident size、admission 标记、heap retouch 等控制面开销，但没有物理阻止低分对象进入 vLLM prefix cache。因此第一版几乎没有减少 cache 登记、eviction 或后续 reorder 压力，只增加了策略 bookkeeping。

主要新增路径在 `h1/sitecustomize.py`：

- `cache_full_blocks(...)`：每个新 full block 推断 object id、创建/更新 profile、登记 block-object 映射、更新 heap rank，并收集 `admission_object_blocks`。
- `_edgekv_evaluate_object_admission(...)`：按 object 计算 accept/reject；第一版 reject 只打标，不阻止 `original_cache_full_blocks(...)`。
- `_edgekv_min_resident_object_score(...)`：对当前 resident profiles 做线性扫描；当前 sharedgpt smoke 约 271 个对象，规模不大但属于新增路径。
- `_edgekv_record_lpe_monitor(...)`：admit/lookup/reorder candidate 路径仍有事件组装和 early-check 开销。
- reject 后还会对 rejected object blocks 做一次 `_edgekv_heap_touch_block(...)`，用于让 rejected object 在同分时优先驱逐。

`free_queue_reorder_time_ms` 不是主要原因：0.75 只从约 3210ms 增到 3236ms，0.825 从约 1598ms 增到 1617ms，远小于总 policy 开销增量。新增开销主要来自 cache/lookup/admission bookkeeping，而不是 reorder 变慢。

## Strict Upgrade Impact

strict 后是否还会出现同类增加，取决于“跳过的缓存工作量”是否大于“新增 admission/splitting 开销”。

可能仍增加的情况：

- reject 数量很少。当前 sharedgpt smoke 的 `admission_rejection_count` 为 0.75: 32/271，0.825: 3/271，0.95: 3/271；在 0.825/0.95 上 strict 能跳过的对象很少。
- strict 实现过重，例如每个 block 都重新扫描 resident profiles、把 `original_cache_full_blocks(...)` 拆成大量小调用、继续记录高频 monitor 事件。
- admission 判断误拒绝后续会复用的对象，导致 hit rate 下降，TTFT 增加会超过第一版的纯控制面开销。

可能改善的情况：

- rejected object 覆盖较多 blocks，strict 能减少 prefix-cache 登记、resident object 数量、后续 eviction 和 heap/reorder 工作。
- low-score object 被正确拒绝后，高价值对象保留时间变长，hit rate 或有效命中位置改善。

第二版实现时必须控制 strict 开销：

- admission decision 只按 object 计算一次并缓存，不按 block 重复扫描。
- mixed accepted/rejected 区间只做最少拆分，避免大量调用 `original_cache_full_blocks(...)`。
- strict 模式默认关闭或降采样高频 monitor 事件，尤其是 reorder candidate 事件。
- 新增 `strict_cached_block_skip_count`、`strict_admission_rejection_count`、`strict_admission_fallback_count`，用 skip 数量判断 strict 是否真的抵消了额外策略成本。
- object 边界不可靠时降级 diagnostic，避免错误拒绝导致 hit rate 和 TTFT 恶化。

结论：第一版变慢是“只加控制面、不减数据面”的预期副作用。strict 升级后不必然变慢，但在 rejection 很少或实现过重时仍可能出现 `policy_time_us_avg` 上升；TTFT 是否上升主要取决于 strict 是否误拒绝可复用对象，以及跳过的缓存/驱逐工作是否足够多。

# 第二版升级计划：Strict Object-Level Admission

## Summary

第二版实现 `H1_LPE_ADMISSION_MODE=strict`。默认仍保持 `diagnostic`，只有显式设置 `strict` 时才对低分对象执行物理拒绝：被拒绝对象不注册进 vLLM prefix cache，但不影响当前请求的正常 prefill/decode，只影响未来复用命中。

## Key Changes

- 保留第一版 score 口径：`score = p_reuse * c_recomp_ms / logical_size_mb`，strict 只改变 admission 行为，不重新定义 score。
- 在 `h1/sitecustomize.py` 的 `cache_full_blocks(...)` 包装层中，把 object profile/admission decision 前移到调用 `original_cache_full_blocks(...)` 之前。
- 当 `EDGEKV_H1_GPU_POLICY=h1_lpe` 且 `H1_LPE_ADMISSION_MODE=strict`：
  - `accept` 对象走原生 `original_cache_full_blocks(...)`。
  - `reject` 对象跳过其完整 object 范围的 prefix-cache 登记。
  - 若一次 `cache_full_blocks` 调用里同时包含 accepted/rejected 对象，则按 object 连续区间拆分，只把 accepted 区间传给原生函数。
  - 若 object 边界无法可靠推断，降级为 diagnostic，不执行物理拒绝。
- 新增 strict 诊断字段：
  - `strict_admission_rejection_count`
  - `strict_cached_block_skip_count`
  - `strict_admission_fallback_count`
  - profile 内记录 `admission_mode=strict`、`strict_cache_skipped=True/False`、`strict_fallback_reason`。
- 第一版 object-level eviction 继续保留：被 strict 拒绝的对象理论上不应成为 resident object；若残留诊断状态存在，也继续按 rejected 优先驱逐。

## Implementation Notes

- strict 的拒绝对象只跳过 prefix-cache 注册，不释放、不改写当前请求持有的 KV block，不直接操作 vLLM block/hash 内部结构。
- 优先实现“整对象 all-or-none admission”：一个 object 的 blocks 要么全部登记，要么全部跳过，避免部分对象缓存导致 LPE 排序再次退化。
- 对非 LPE 策略、`diagnostic` 模式、无 object profile、pinned object、无法拆分的混合区间，保持现有行为。
- 对 `H1_LPE_ADMISSION_MODE=strict` 的 env 语义改为真实生效；非法值继续回落到 `diagnostic`。

## Test Plan

- 单元测试：
  - diagnostic 模式行为完全不变，reject 仍会调用原生 cache 登记。
  - strict 模式下低分 object 不调用原生 cache 登记，stats 记录 skipped blocks。
  - strict 模式下高分 object 原样调用原生 cache 登记。
  - mixed accepted/rejected object 输入会只登记 accepted object 区间。
  - 无法可靠推断 object 边界时降级 diagnostic，并记录 fallback。
  - LRU/LFU/default 策略不受 `H1_LPE_ADMISSION_MODE=strict` 影响。
- 回归检查：

```bash
python3 -m py_compile h1/sitecustomize.py h1/test_h1_rag_trace.py
```

- 真机 smoke：
  - 小规模 `sharedgpt_v5` 和 `hotqa_ws2` 各跑一次 `diagnostic` 与 `strict`。
  - 验证无 vLLM prefix cache 异常、无 hash/block 状态错误。
  - 对比 `hit rate`、`TTFT`、`admission_rejection_rate`、`strict_cached_block_skip_count`。
  - strict 预期：低价值对象缓存占用下降；hit 不应因误拒绝高分对象明显恶化；TTFT 不应出现异常长尾。

## Assumptions

- 第二版只实现 opt-in strict，不改变默认实验行为。
- strict admission 的物理含义是“不注册到 prefix cache”，不是阻止当前请求计算 KV。
- object 边界不可靠时优先保证正确性，降级 diagnostic，而不是冒险拆分 vLLM 内部 cache 调用。
- 第一版稳定标准是：score 分布正常、diagnostic rejection 统计可信、object-level eviction 无 vLLM 状态错误。
