# H1 实验框架重构方案（思路篇）

## 一、为什么要重构？—— 现状痛点分析

当前代码库在支撑实验快速迭代方面已经暴露出明显的结构性瓶颈，主要体现在以下四个方面：

### 1. 上帝对象泛滥，职责高度耦合
`h1/sitecustomize.py` 是一个超过 700 行的“巨无霸”文件，它同时承担了：
- 策略逻辑（`_edgekv_rank_tuple` 中的 LRU/LFU/LPE 分支判断）
- 状态管理（freq、recency、score 的维护）
- 统计收集（`_edgekv_note_gpu_stat` 散布各处）
- vLLM 内部钩子注入（monkey-patch `BlockPool`）
- 环境变量解析（20+ 个 `EDGEKV_*` 变量）

**后果**：改动一个策略的排名逻辑，可能波及统计代码；新增一个统计指标，需要在策略代码中到处插入埋点。

### 2. 开闭原则被违反，扩展成本极高
新增一个策略（例如 `h1_arc`）当前需要修改 `_edgekv_rank_tuple` 中的 `if` 分支，涉及至少 3 个不同函数的改动。这导致每增加一个新策略，核心逻辑的稳定性都会受到冲击，回归测试成本线性增长。

### 3. 配置管理“三足鼎立”
环境变量（`EDGEKV_*`）、CLI 参数（argparse）、文件顶部硬编码（`MAX_REQUESTS=1024`）三者并存。一个实验参数（如并发度）可能同时出现在三个地方，修改时极易遗漏，导致实验结论前后不一致。

### 4. 实验编排代码大量重复
`run_step3_budget_tiers.py`、`run_step3_repeat.py`、`run_step3_real.py` 三个文件的循环骨架几乎相同（遍历 budget/policy/rep），核心差异仅在于“内层如何构建参数”和“最后如何汇总”。每次新增实验类型（如 `step4_throughput`）都意味着复制粘贴一大段代码。

## 二、重构的指导思想（SOLID 原则映射）

| SOLID 原则 | 在本次重构中的具体体现 |
|------------|------------------------|
| **单一职责（SRP）** | 策略逻辑、统计收集、钩子注入、实验编排、配置解析严格物理隔离为独立模块 |
| **开闭原则（OCP）** | 新增策略/统计输出/实验类型时，只增新类，不修改现有核心类 |
| **里氏替换（LSP）** | 所有策略实现统一的 `CachePolicy` 接口，运行器无需知道具体策略类型 |
| **接口隔离（ISP）** | `CachePolicy` 只暴露 `on_access/rank/on_evict`，统计接口只暴露 `emit`，不强迫依赖不用的方法 |
| **依赖倒置（DIP）** | 高层运行器依赖 `CachePolicy` 接口，而非具体策略类；vLLM 钩子依赖统计接口而非具体导出实现 |

## 三、核心设计模式应用思路

### 🔹 策略模式 + 工厂模式 —— 解决“策略硬编码”
将现有的 `_edgekv_rank_tuple` 中的 `if policy == ...` 分支拆解为独立的策略类（`LRUPolicy`、`LFUPolicy`、`LPEPolicy`），每个类只负责维护自己的状态和排名逻辑。通过一个策略工厂，根据字符串名称（如 `h1_lpe`）动态返回实例。**核心收益**：新增策略只需新建一个类并注册，核心钩子代码零修改。

### 🔹 模板方法模式 —— 解决“编排代码重复”
将三个 `run_step3_*.py` 中的公共循环逻辑上提到一个抽象基类 `ExperimentRunner` 中，定义好 `run()` 的骨架流程（准备 trace → 循环执行 cell → 汇总 → 清理），将差异部分（如何构建 cell 参数、如何汇总）作为抽象方法留给子类实现。**核心收益**：新增实验类型时，只需继承基类并重写 2-3 个方法，代码量减少 70%。

### 🔹 观察者模式 —— 解决“统计与业务耦合”
将散布在策略代码中的统计埋点（`_edgekv_note_gpu_stat`）替换为向一个“事件总线”发射事件（如 `evict`、`hit`、`admit`）。各种输出方式（JSON 文件、控制台打印、Prometheus）均实现为订阅者，在系统启动时注册。**核心收益**：策略层彻底忘记统计的存在，新增统计输出无需改动策略代码。

### 🔹 建造者模式 —— 解决“配置散乱”
设计一个 `ExperimentConfig` 配置对象，通过建造者链式构造：YAML 默认值 → 环境变量覆盖 → CLI 参数覆盖 → 最终冻结。所有底层函数（包括 `run_cell`）只接受这个配置对象。**核心收益**：配置来源统一、优先级清晰，参数修改只需改 YAML 或 CLI。

### 🔹 依赖注入 + 适配器模式 —— 解决“隐式副作用”
将 `sitecustomize.py` 的自动执行改为显式安装。在 `run_h1_vllm0110_real.py` 的入口处，**显式组装**所有依赖：
1. 从工厂创建策略实例
2. 创建统计收集器并挂载订阅者
3. 将策略和统计注入到 `VLLMInjector`（适配器）
4. 调用 `injector.install()` 应用 monkey-patch

这样，`sitecustomize` 瘦身为只保留兼容性补丁，不再包含任何策略逻辑。**核心收益**：消除“导入即副作用”，单元测试时可以注入 Mock 对象，彻底解决测试难的问题。

### 🔹 命令模式 —— 解决“入口混乱”
将多个 `run_*.py` 脚本合并为一个统一 CLI 入口（如 `h1cli`），每个子命令（`step3 repeat`、`step3 real`、`find interval`、`quick-test`）对应一个命令类。**核心收益**：用户只需学习一个命令的用法，新实验类型只需新增命令类，无侵入。

## 四、重构后的目标架构

```text
h1/
├── core/              # 抽象接口与公共基类（策略接口、配置对象、运行器基类、统计接口）
├── policies/          # 具体策略实现（lru.py / lfu.py / lpe.py）+ 工厂
├── stats/             # 统计收集器 + 各种导出器（JSON / Console / Prometheus）
├── hooks/             # 唯一的 vLLM 钩子注入点（适配器），依赖策略和统计接口
├── runners/           # 具体实验运行器（Step3Tiers / Step3Repeat / Step3Real）
├── cli/               # 统一命令行入口 + 命令类
├── summarizers/       # 汇总工具（读取数据对象，不直接读 CSV）
├── visualizers/       # 绘图工具
├── validators/        # D3 / D4 验证
└── sitecustomize.py   # 瘦身版（仅保留兼容补丁，策略/统计完全移除）
```

**关键设计约束**：
- `core/` 不依赖任何外部具体实现（vLLM、PyTorch），只定义接口。
- `hooks/` 是**唯一**接触 vLLM 内部 API 的地方，方便 vLLM 版本升级时集中修改。
- 策略类和统计导出器之间**零耦合**，通过事件总线间接通信。

## 五、重构后的典型交互流程（新架构如何运行）

```text
1. 用户执行: h1cli step3 real --budgets tight mid --policies h1_lpe

2. CLI 解析 → 创建 Step3RealCommand 对象
      → 通过 Builder 构建 ExperimentConfig（合并 YAML + CLI 覆盖）

3. 创建 Step3RealRunner（继承 ExperimentRunner）
      → 调用 runner.run(config)
            ├── _prepare_trace() → 必要时调用 build_h0_replay_trace.py
            ├── for rep in reps:
            │   for budget in budgets:
            │   for policy in policies:
            │       cell_dir = ...
            │       policy_obj = PolicyFactory.create(policy)
            │       stats = StatsCollector()
            │       stats.subscribe(JSONExporter(cell_dir))
            │       injector = VLLMInjector(policy_obj, stats)
            │       injector.install()   # 此时才应用 monkey-patch
            │       run_single_cell(...)
            │       injector.uninstall()
            ├── _aggregate() → 调用 summarizer
            └── _cleanup()
```

**与旧流程的关键差异**：
- 策略对象是**显式创建**的，而不是通过环境变量隐式读取。
- 统计收集器是**显式挂载**的，而不是在策略代码中硬编码。
- monkey-patch 是**按需安装**的，而不是导入即生效。

## 六、分阶段实施路线图（风险可控的渐进式重构）

| 阶段 | 目标 | 工作内容 | 风险等级 | 验收标准 |
|------|------|---------|---------|----------|
| **第一阶段：接口抽象-完成** | 逻辑隔离，行为不变 | 定义 `CachePolicy` 接口；将 `_edgekv_rank_tuple` 的策略分支逻辑按策略拆解到独立类中；`sitecustomize` 内部改为通过工厂获取策略实例并委托调用 | 🟢 低 | 现有所有 `run_*.py` 输出结果与重构前完全一致 |
| **第二阶段：配置统一** | 消除硬编码和散乱的配置 | 引入 `ExperimentConfig` 建造者；先在 `run_test.py` 试点，替换顶部硬编码；逐步迁移其他启动器 | 🟡 中 | 所有实验配置可通过 YAML + CLI 覆盖，不再依赖文件顶部常量 |
| **第三阶段：运行器模板化** | 消除编排代码重复 | 创建 `ExperimentRunner` 抽象基类，将三个 `run_step3_*.py` 重构为子类 | 🟡 中 | 三个脚本的代码量总计减少 60% 以上 |
| **第四阶段：统计与依赖解耦** | 消除隐式副作用 | 将 `_edgekv_note_gpu_stat` 替换为 `StatsCollector` 事件发射；移除 `sitecustomize` 的自动执行，改为入口显式安装钩子 | 🔴 高（需仔细验证） | 单元测试可不启动 vLLM 而独立测试策略逻辑；统计输出可通过订阅者灵活扩展 |
| **第五阶段：统一 CLI** | 消除入口混乱 | 实现 `h1cli` 统一命令入口，旧 `run_*.py` 转为薄包装器并给出 DeprecationWarning | 🟢 低 | 所有用户命令可通过 `h1cli --help` 查看 |
| **第六阶段：汇总/可视化数据化** | 提高健壮性 | 将 `summarize_*.py` 和 `visualize_*.py` 改为接收数据对象而非直接读 CSV/JSON | 🟢 低 | 汇总和绘图可独立于文件系统进行单元测试 |

**实施建议**：第一阶段和第二阶段可以并行推进，因为它们互不依赖；第三阶段必须在第二阶段完成后进行；第四阶段是核心突破点，建议单独安排一个迭代周期重点攻关，并准备充分的回归测试用例。

## 七、预期收益总结

| 维度 | 重构前 | 重构后 |
|------|--------|--------|
| **新增策略成本** | 修改核心函数 3+ 处，约 2 小时 | 新建类 + 注册，约 15 分钟 |
| **核心逻辑可测试性** | 几乎为 0（需真实 vLLM 环境） | 策略类可独立单测，覆盖率 > 70% |
| **实验编排代码量** | 三个文件合计约 600 行重复 | 基类 150 行 + 子类各 80 行 |
| **配置变更** | 改代码 + 环境变量 + CLI | 只改 YAML 或 CLI |
| **统计输出扩展** | 需修改策略代码中的埋点 | 新增订阅者类，零侵入 |
| **vLLM 版本升级影响** | 需在 sitecustomize 中多处查找修改 | 只需修改 hooks/ 适配器 |

---

## 八、结语

本重构方案的核心思想是 **“将变化封装在接口之后，将组合放在入口之前”**。通过合理的抽象和设计模式应用，将当前高度耦合的“脚本集合”重塑为一个**面向接口、依赖显式、配置驱动**的微内核框架。重构完成后，团队将能够以极低的成本快速迭代新策略、新实验场景和新分析需求，真正实现“对扩展开放，对修改关闭”。建议按阶段逐步推进，每个阶段都保持系统可运行，确保实验数据零丢失、结论可复现。
