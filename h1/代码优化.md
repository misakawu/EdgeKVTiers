# 代码优化思路
---

## 一、优化总纲：从“脚本集合”进化为“可插拔框架”

**核心理念**：将 H1 从一个由多个独立 `run_*.py` 脚本拼接而成的“脚本集合”，重构为一个**面向接口、依赖显式、配置驱动**的微内核架构。

**三条黄金原则**：
1. **开闭原则（OCP）**：新增策略、预算、统计输出时，只增新类，不修改旧代码。
2. **单一职责（SRP）**：策略逻辑、vLLM 钩子、统计收集、实验编排、配置解析，严格物理隔离。
3. **显式优于隐式**：消除 `sitecustomize.py` 的全局自动副作用，所有扩展点通过入口显式组装。

---

## 二、痛点对应的专项优化思路（按设计模式分类）

### 1. 策略逻辑剥离（策略模式 + 工厂模式）
**现状**：`sitecustomize.py` 中充斥着 `if policy == "h1_lru"` 的分支，新增一个策略（比如 `h1_arc`）需要修改核心函数，违反 OCP。

**优化思路**：
- 定义统一的 **`CachePolicy` 接口**（抽象类），约定 `on_access(block)`、`rank(block)`、`on_evict(block)` 等生命周期方法。
- 将 `LRU`、`LFU`、`LPE` 各自实现为独立类，各自维护内部状态（频率、分数、近因）。
- 建立一个**策略注册表（工厂）**，通过策略名字符串（如 `h1_lpe`）动态返回实例。
- vLLM 钩子层只持有 `CachePolicy` 接口引用，永远不依赖具体策略类。

**收益**：新增策略 = 新建一个类 + 在启动配置中注册，核心引擎和钩子代码零修改。

### 2. 统一配置管理（建造者模式 + 外观模式）
**现状**：配置散落在环境变量（`EDGEKV_*`）、CLI 参数（`argparse`）、脚本顶部硬编码（`DEFAULT_*`）三处，修改一个参数要同步改三个地方。

**优化思路**：
- 设计一个 **`ExperimentConfig`** 配置对象，作为全系统唯一的配置外观（Facade）。
- 引入**建造者（Builder）** 链式构造配置：默认值从 YAML 文件加载 → 被环境变量覆盖 → 被 CLI 参数覆盖 → 最终冻结为不可变对象。
- CLI 参数大幅精简，只保留最常用的 `--config`（指定 YAML 路径）和 `--override`（键值对覆盖）。
- 所有底层函数（如 `run_cell`）只接受 `ExperimentConfig` 对象，不再传递长参数列表。

**收益**：配置来源统一、优先级清晰、易于序列化和版本控制。

### 3. 实验流程复用（模板方法模式）
**现状**：`run_step3_budget_tiers.py`、`run_step3_repeat.py`、`run_step3_real.py` 存在大量重复循环（遍历预算、策略、重次），只有“内层调用”和“汇总方式”略有不同。

**优化思路**：
- 设计一个抽象基类 **`ExperimentRunner`**（模板），定义 `run()` 的骨架算法：
  1. 加载/生成 Trace（钩子）
  2. 嵌套循环（预算 × 策略 × Reps）（钩子）
  3. 执行单 Cell（公共方法）
  4. 汇总结果（钩子）
  5. 清理中间文件（钩子）
- 三个具体场景分别继承该基类，只需重写：
  - `_should_skip_cell()`（是否已存在且不强制）
  - `_get_cell_params()`（构建命令行参数）
  - `_aggregate()`（调用对应的 summarize 脚本）

**收益**：消除重复代码，新增实验类型（如 `step4_throughput`）工作量降低 80%。

### 4. 统计收集解耦（观察者模式 + 发布-订阅）
**现状**：`_edgekv_note_gpu_stat()` 散落在 `sitecustomize` 各处，与策略代码紧耦合，想增加 Prometheus 输出或实时控制台打印必须改动策略源码。

**优化思路**：
- 定义 **`StatsCollector`**（统计收集器）作为事件总线。
- 策略层在关键事件（`evict`、`hit`、`admit`、`reorder`）发生时，向收集器发射事件（携带数据）。
- 各种输出方式（JSON 导出器、CSV 导出器、控制台打印、Prometheus 推送）均实现为**订阅者（Observer）**，在系统启动时注册到收集器。
- 收集器本身不关心输出格式，只负责分发事件。

**收益**：策略层彻底忘记统计的存在，统计扩展（比如要加一个实时 Dashboard）只需新增订阅者，无需触碰核心逻辑。

### 5. 依赖注入 + 显式组装（控制反转原则）
**现状**：`run_h1_vllm0110_real.py` 通过 `import sitecustomize` 隐式触发全局 monkey-patch，策略由环境变量在运行时被动读取，导致单元测试极难编写（必须真实拉起 vLLM）。

**优化思路**：
- 将 `sitecustomize.py` 瘦身为纯粹的 **“vLLM 钩子定义库”**，不再自动执行。
- 在 `run_h1_vllm0110_real.py` 的 `main()` 入口显式完成“依赖组装（DI Container）”：
  1. 解析配置得到策略名。
  2. 从工厂创建策略实例。
  3. 创建统计收集器并挂载订阅者。
  4. 将策略实例和统计收集器注入到 `VLLMHookInstaller`。
  5. 调用 `installer.install()` 应用 monkey-patch（此时才生效）。
- 单元测试时，可以传入 Mock 策略和 Mock 统计，完全不依赖真实 vLLM 环境。

**收益**：依赖关系完全可视化，测试覆盖率可大幅提升，且消除了“导入即副作用”的反模式。

### 6. 统一入口与命令路由（命令模式）
**现状**：根目录下散落着 `run_step3_real.py`、`run_find_interval_2.py`、`run_test.py` 等多个入口，用户困惑该用哪个。

**优化思路**：
- 设立唯一的命令行入口（如 `h1cli` 或 `python -m h1.cli`）。
- 将每一个实验类型（`step3-repeat`、`step3-real`、`find-interval`、`quick-test`）封装为一个**命令对象（Command）**，实现统一的 `execute(args)` 接口。
- 主解析器根据第一个子命令参数路由到对应命令对象执行。
- 每个命令对象内部再调用对应的 `ExperimentRunner` 子类。

**收益**：用户只需学习一个命令的用法（`h1cli <subcommand> --help`），扩展新实验类型只需新增命令类，修改无侵入。

## 三、架构蓝图：新的模块划分（逻辑视图）

- **`h1.core`**：定义核心抽象（策略接口、配置对象、统计接口、运行器基类），不依赖任何外部具体实现。
- **`h1.policies`**：存放所有策略具体实现（`lru.py`、`lfu.py`、`lpe.py`），依赖 `core` 中的接口。
- **`h1.stats`**：统计收集器及各种导出器（`json_exporter.py`、`console_reporter.py`），仅依赖 `core` 中的事件定义。
- **`h1.hooks`**：**唯一一处**接触 vLLM 内部 API 的模块，负责将策略接口适配到 `BlockPool` 的具体回调，执行依赖注入。
- **`h1.runners`**：各种实验运行器实现（`step3_tiers.py`、`step3_repeat.py`、`quick_test.py`），继承 `core.ExperimentRunner`。
- **`h1.cli`**：统一命令行入口，包含所有命令对象。
- **`h1.configs`**：存放默认的 YAML 配置文件（如 `default.yaml`、`tight_budget.yaml`），供建造者加载。
- **`h1.compat`**：过渡期保留的旧版脚本（`run_*.py`），内部转调新架构并打印 DeprecationWarning。

## 四、分阶段演进路线图（风险可控的渐进式重构）

### 第一阶段：接口抽象（零功能变化）
- 将 `sitecustomize` 中的 `_edgekv_rank_tuple` 等核心函数**原样复制**到新策略类中（不改变逻辑，只做物理移动）。
- `sitecustomize` 内部改为通过工厂获取策略实例并委托调用，外部行为完全不变。
- 此时已完成逻辑隔离，但运行路径仍是旧的，风险最低。

### 第二阶段：配置与入口整合
- 引入 `ExperimentConfig` 建造者，首先在 `run_test.py` 试点，替换顶部硬编码。
- 逐步将 `run_step3_*.py` 改造为接受配置对象，而非 `argparse.Namespace`。
- 新增 `h1cli` 统一入口，旧脚本保留但标记为过时。

### 第三阶段：运行器模板化
- 创建 `ExperimentRunner` 抽象基类，将三个 `run_step3_*.py` 的公共循环逻辑上提。
- 子类仅保留参数构建和汇总调用的差异代码。

### 第四阶段：统计与依赖解耦
- 将 `_edgekv_note_gpu_stat` 替换为 `StatsCollector.emit()` 调用。
- 移除 `sitecustomize` 的自动执行，改为在 `run_h1_vllm0110_real.py` 入口显式安装钩子。

### 第五阶段：全面切换与清理
- 所有旧版 `run_*.py` 脚本删除或转为仅执行新 CLI 的薄包装器。
- 文档同步更新，所有示例命令切换到 `h1cli` 格式。

## 五、优化后的预期收益（定性 + 定量）

| 维度 | 优化前 | 优化后 |
|------|--------|--------|
| **新增策略耗时** | 需改动 `sitecustomize` 核心函数，至少 2 小时 | 新建类 + 注册，约 15 分钟 |
| **单元测试覆盖** | 几乎为 0（因依赖 vLLK 环境） | 策略层、配置层、运行器层均可单测，覆盖 > 70% |
| **实验脚本代码量** | 三个脚本合计 ~600 行重复代码 | 公共基类 ~150 行，子类每个 ~80 行 |
| **配置变更** | 需改代码 + 环境变量 | 只改 YAML 文件或 CLI 覆盖 |
| **调试体验** | 调用栈跨越 5 层文件，难以追踪 | 依赖显式，入口清晰，可本地断点调试 |

## 六、特别注意的非功能性保障

1. **性能零退化**：策略接口的方法调用在热路径（每次缓存访问）上，需确保使用 `__slots__` 和避免不必要的对象分配。抽象接口仅用于编译时检查，运行时仍保持直接方法调用。
2. **向后兼容**：保留旧版脚本至少两个大版本，内部转调用新核心并输出警告，给团队成员充分的迁移缓冲期。
3. **配置迁移工具**：提供一个辅助脚本，将旧版的环境变量/CLI 参数自动转写成 YAML 配置文件，降低学习成本。

这套优化方案的核心思想是 **“将变化封装在接口之后，将组合放在入口之前”**，完全符合“溜达原则”（SOLID）的精髓。若后续需要扩展到 H0 或 script 目录，同样的架构思路可平滑复制——将 `run_h0_vllm.py` 中的 trace 构建逻辑作为“数据源策略”注入到同样的运行器框架中，实现 H0/H1 实验的完全统一。
