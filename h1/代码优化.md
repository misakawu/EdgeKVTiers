# 代码审查与基于 SOLID 的重构方案

## 一、代码审查总览

本项目是一个针对 vLLM GPU 前缀缓存（Prefix Caching）的 **KV 缓存策略实验平台**，核心目标是比较 LRU、LFU 和自定义 LPE（基于对象级收益/代价）策略在不同 GPU 内存预算下的性能差异（命中率、延迟、吞吐量）。代码库分为 `h0/`、`h1/`、`scripts/` 三大模块，并包含共享工具 `edgekv_cop.py`。

### 1.1 文件职责梳理

| 模块 | 主要文件 | 职责 |
|------|---------|------|
| **h0/** | `build_h0_replay_trace.py`, `run_h0_vllm.py`, `run_h0_hetero_probe.py`, `sitecustomize.py` (兼容性补丁) | 构建 replay trace、运行早期 vLLM 实验、异质探针分析 |
| **h1/** | `run_h1_vllm0110_real.py`：核心实验引擎 | 加载 trace，创建 LLM 实例，运行策略实验，输出 summary/requests |
| | `sitecustomize.py`：**核心运行时补丁（约 700 行）** | 通过 monkey‑patch 钩入 vLLM `BlockPool`，实现策略逻辑、状态维护、统计收集、自由队列重排 |
| | `run_step3_*.py`（三个文件）：实验编排 | 循环不同 budget/policy/rep，调用 `_runner.run_real_cell()` 执行单 cell |
| | `run_find_*.py`：定档/找曲线 | 扫描 budget 区间，根据命中率自动选择档位或判断曲线平滑度 |
| | `summarize_*.py`, `visualize_*.py`：汇总与绘图 | 读取各 cell 的 summary JSON，计算中位数，生成 CSV 和 PNG/PDF 图表 |
| | `_runner.py`：共享辅助函数 | 执行子进程、清理、调用汇总脚本、运行 D3 验证 |
| | `test_h1_rag_trace.py`：单元测试 | 验证 trace 加载、侧车字段、策略钩子行为 |
| **scripts/** | `optimize_h0_pressure_trace.py`, `hotqa_jsonl_generation.py`, `sharedgpt_jsonl_generation.py` | 生成具有可控重复模式的 JSONL trace |
| **根目录** | `edgekv_cop.py` | 对象级 COP 分析器，用于估算 p_reuse、c_recomp、score |

### 1.2 核心问题总结

#### 🔴 **设计缺陷**

- **`sitecustomize.py` 是“上帝对象”**  
  它集策略逻辑（`_edgekv_rank_tuple`）、状态管理（freq/recency/score）、统计收集（`_edgekv_note_gpu_stat`）、vLLK 钩子注入、环境变量解析于一身。**违反单一职责（SRP）**。

- **策略实现硬编码**  
  `_edgekv_rank_tuple` 通过 `if policy == 'h1_lru'` 等分支决定行为。新增策略需修改核心函数，**违反开闭原则（OCP）**。

- **配置散乱**  
  环境变量（`EDGEKV_*`）、CLI 参数（argparse）、硬编码常量（如 `run_step3_budget_tiers.py` 顶部的 `MAX_REQUESTS = 1024`）三处并存，修改一处需同步多处，易出错。

- **实验编排代码重复**  
  `run_step3_budget_tiers.py`、`run_step3_repeat.py`、`run_step3_real.py` 的循环骨架几乎相同，仅内部参数和汇总方式不同，大量复制粘贴。

#### 🟡 **可维护性问题**

- **隐式副作用**  
  `sitecustomize.py` 在导入时自动执行 `_install_edgekv_gpu_prefix_cache_patch()`，导致无法在不打补丁的情况下导入模块（单元测试困难）。策略通过环境变量被动读取，依赖隐式。

- **统计与业务耦合**  
  统计函数（如 `_edgekv_note_gpu_stat`）直接散布在策略逻辑中，新增输出（如 Prometheus）需修改所有统计点。

- **文件名与职责模糊**  
  `_runner.py` 包含了 `validate_d3_for_lpe_cells` 等上层业务，而不仅仅是“底层运行器”，职责边界不清。

#### 🟠 **可测试性差**

- 核心策略逻辑无法独立测试（必须启动 vLLM 或模拟大量内部对象）。
- 单元测试仅覆盖少量表面功能，缺少对 `_edgekv_rank_tuple`、`_edgekv_reorder_free_queue` 等关键函数的隔离测试。

#### 🔵 **扩展性瓶颈**

- 增加新策略需要改动至少三个文件（`sitecustomize` 的 rank 逻辑、策略工厂、环境变量判断）。
- 增加新实验类型（如 `step4_throughput`）需要复制现有编排脚本。

## 二、基于 SOLID 的重构方案（设计模式驱动）

### 2.1 总体目标

将代码从“脚本集合”重构为 **可插拔的微内核框架**，核心原则：
- **单一职责**：每个模块只做一件事。
- **开闭原则**：新增策略、实验类型、统计输出无需修改现有核心代码。
- **依赖倒置**：高层模块依赖抽象接口，不依赖具体实现。
- **显式优于隐式**：消除自动副作用，所有依赖通过入口显式组装。

### 2.2 模块重新划分

```
h1/
├── core/
│   ├── interfaces.py          # CachePolicy, StatsCollector, ExperimentRunner 接口
│   └── config.py              # ExperimentConfig 类 + Builder
├── policies/
│   ├── base.py                # 抽象基类 CachePolicy
│   ├── lru.py, lfu.py, lpe.py # 具体策略
│   └── factory.py             # 策略工厂
├── stats/
│   ├── collector.py           # StatsCollector（事件总线）
│   ├── events.py              # 事件类型定义
│   └── exporters/             # JSONExporter, ConsoleReporter, PrometheusExporter
├── hooks/
│   └── vllm_injector.py       # 唯一的 vLLM monkey-patch 入口（依赖策略和统计接口）
├── runners/
│   ├── base.py                # ExperimentRunner 模板
│   ├── step3_tiers.py
│   ├── step3_repeat.py
│   └── step3_real.py
├── cli/
│   ├── main.py                # 统一命令行入口（命令模式）
│   └── commands/              # Step3Command, FindIntervalCommand, QuickTestCommand
├── summarizers/               # 汇总工具（读取数据对象，不直接读 CSV）
├── visualizers/               # 绘图工具（接收数据对象）
├── validators/                # D3, D4 验证
├── compat/                    # 过渡期旧脚本（内部转调新架构并给出 DeprecationWarning）
└── sitecustomize.py           # 瘦身版（仅保留兼容补丁，实际策略委托给 hooks）
```

### 2.3 关键设计模式应用

#### 🔸 **策略模式 + 工厂模式**
- 定义 `CachePolicy` 接口：`on_access(block)`, `rank(block)`, `on_evict(block)`。
- 每个策略独立类（`LRUPolicy`, `LFUPolicy`, `LPEPolicy`），各自维护内部状态（freq, recency, score）。
- 工厂根据名称（`h1_lru`）创建实例。
- **收益**：新增策略 = 新建类 + 注册，核心钩子零修改。

#### 🔸 **建造者模式**
- 使用 `ExperimentConfig.Builder` 链式构建配置，支持从 YAML/JSON 加载默认值，再被 CLI 和环境变量覆盖。
- 所有实验脚本接收一个 `ExperimentConfig` 对象，不再有长参数列表。

#### 🔸 **模板方法模式**
- `ExperimentRunner` 抽象基类定义 `run()` 骨架：
  1. `_prepare_trace()`（钩子）
  2. `_run_cells()`（钩子，核心循环）
  3. `_aggregate()`（钩子）
  4. `_cleanup()`（钩子）
- 子类（如 `Step3TiersRunner`）只重写这些钩子，复用公共逻辑。

#### 🔸 **观察者模式**
- `StatsCollector` 作为事件总线，策略层在关键动作（`evict`, `hit`, `admit`, `reorder`）时调用 `collector.emit(event, data)`。
- 各种输出（JSON、CSV、控制台、Prometheus）实现为订阅者，在启动时注册到收集器。
- **收益**：策略完全不知道统计细节，扩展统计输出无需改动策略代码。

#### 🔸 **依赖注入**
- 在 `main()` 中显式组装依赖链：创建策略 → 创建统计收集器 → 挂载订阅者 → 创建 `VLLMInjector` 并注入策略和统计 → 调用 `injector.install()` 应用补丁。
- 完全消除 `sitecustomize` 的自动副作用，单元测试时可注入 Mock 策略和 Mock 统计。

#### 🔸 **命令模式**
- 单一 CLI 入口 `h1cli`，子命令如 `step3 repeat`, `step3 real`, `find interval`, `quick-test`。
- 每个子命令对应一个 `Command` 类，解析参数并调用对应运行器。
- **收益**：用户只需学习一个命令，新实验类型只需新增 Command 子类。

#### 🔸 **适配器模式**
- `VLLMInjector` 作为适配器，将 `CachePolicy` 接口适配到 vLLM 的 `BlockPool` 回调（`get_cached_block`, `cache_full_blocks`, `_maybe_evict_cached_block`）。
- 这是**唯一一处**与 vLLM 内部 API 耦合的代码，便于升级 vLLM 时集中修改。

### 2.4 重构后的典型调用流程

```text
用户执行: h1cli step3 real --budgets tight mid --policies h1_lpe

CLI 解析 → 创建 Step3RealCommand
    → 构建 ExperimentConfig (Builder 从 YAML + CLI 覆盖)
    → 创建 Step3RealRunner (继承 ExperimentRunner)
    → 调用 runner.run(config)
        ├── _prepare_trace() → 调用 build_h0_replay_trace.py（若需要）
        ├── for rep in reps:
        │   for budget in budgets:
        │   for policy in policies:
        │       cell_dir = ...
        │       policy_obj = PolicyFactory.create(policy)
        │       stats = StatsCollector()
        │       stats.subscribe(JSONExporter(cell_dir))
        │       injector = VLLMInjector(policy_obj, stats)
        │       injector.install()  # 应用 monkey-patch
        │       runner.run_single_cell(cell_dir, policy, budget, config)
        │       injector.uninstall() # 恢复（可选）
        ├── _aggregate() → 调用 summarizer 生成汇总 CSV
        └── _cleanup() → 删除中间文件（除非 --keep-cells）
```

### 2.5 分阶段演进路线

| 阶段 | 工作内容 | 风险 | 收益 |
|------|---------|------|------|
| **1. 接口抽象** | 定义 `CachePolicy` 接口，将 `sitecustomize` 中的 rank 逻辑按策略拆解到独立类中，`sitecustomize` 内部改为委托调用（行为不变） | 低 | 逻辑隔离，为后续打基础 |
| **2. 配置统一** | 引入 `ExperimentConfig` Builder，先在 `run_test.py` 试点，替换硬编码；逐步迁移其他 `run_*.py` | 中 | 配置管理清晰，支持 YAML |
| **3. 运行器模板** | 创建 `ExperimentRunner` 基类，将三个 `run_step3_*.py` 重构为子类 | 中 | 消除重复代码，统一骨架 |
| **4. 统计解耦** | 将 `_edgekv_note_gpu_stat` 替换为 `StatsCollector` 事件发射，实现 JSON/控制台导出器 | 中 | 统计与业务解耦，易于扩展 |
| **5. 依赖注入与钩子瘦身** | 移除 `sitecustomize` 的自动安装，改为在 `run_h1_vllm0110_real.py` 入口显式组装并安装钩子；`sitecustomize` 只保留兼容补丁 | 高 | 彻底消除隐式副作用，可测试性大幅提升 |
| **6. 统一 CLI** | 实现 `h1cli` 命令入口，将旧 `run_*.py` 转为薄包装（DeprecationWarning） | 低 | 用户入口唯一化，降低认知负担 |
| **7. 汇总/可视化重构** | 将 `summarize_*.py` 和 `visualize_*.py` 改为接收数据对象而非直接读 CSV，提高灵活性 | 低 | 更健壮，便于单元测试 |

### 2.6 预期收益

| 维度 | 重构前 | 重构后 |
|------|--------|--------|
| **新增策略** | 需修改 `sitecustomize` 核心函数（~3 处），约 2 小时 | 新建类 + 注册，约 15 分钟 |
| **单元测试覆盖** | 核心策略逻辑覆盖 < 10% | 策略类、配置类、运行器均可独立测试，覆盖 > 70% |
| **代码重复** | 三个 `run_step3_*.py` 约 600 行重复循环 | 模板类 ~150 行，子类每个 ~80 行 |
| **配置变更** | 需改代码 + 环境变量 + CLI | 只改 YAML 或 CLI 覆盖 |
| **调试体验** | 调用栈跨越 5 层，依赖隐式 | 依赖显式，入口清晰，可本地断点调试 |
| **统计输出扩展** | 需修改 `sitecustomize` 中的统计点 | 只需新增订阅者类 |

## 三、小结

本项目具备扎实的实验基础，但当前架构在可维护性、可扩展性和可测试性上存在显著短板。通过**策略模式、建造者模式、模板方法、观察者模式、依赖注入和命令模式**等设计手段，可以将其重构为一个高度模块化、符合 SOLID 原则的实验平台。建议按照上述六阶段路线图渐进实施，每阶段都保持系统可运行，降低回归风险。重构完成后，团队将能更高效地迭代新策略、新实验类型和新的统计需求，同时大幅提升代码质量和团队协作效率。
