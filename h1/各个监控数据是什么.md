# D3 数据监控

## n_tokens_p95 结论

当前 `out` 中 stats JSON 的 `n_tokens_p95` 几乎全为 0，主因不是 workload 的 token 数为 0，而是这个字段只从 LPE object profile 聚合得到。多数 `edgekv_gpu_stats_*.json` 对应的进程没有产生 LPE profile，因此 `lpe_profile_count=0`，`n_tokens_min/n_tokens_p50/n_tokens_p95` 都会按默认值写为 0。

已抽样统计现有输出：

- stats JSON 总数：130
- `lpe_profile_count > 0`：10
- `n_tokens_p95 > 0`：10
- 有 lookup/admission 但 `n_tokens_p95=0` 的典型情况：`h1_lru`。LRU 路径只记录 cache lookup/admission/eviction 等诊断计数，不创建 LPE object profile。

因此 `n_tokens_p95=0` 的含义通常是“该进程没有可聚合的 LPE profile”，不是“请求长度为 0”。

## n_tokens_p95 的代码来源

`n_tokens_p95` 在 `h1/sitecustomize.py:get_edgekv_gpu_cache_stats()` 中生成：

```python
profiles = list(_EDGEKV_GPU_LPE_PROFILES.values())
n_tokens_values = [float(profile.get('n_tokens', 0) or 0) for profile in profiles]
stats['n_tokens_min'] = min(n_tokens_values) if n_tokens_values else 0.0
stats['n_tokens_p50'] = _edgekv_percentile(n_tokens_values, 0.50)
stats['n_tokens_p95'] = _edgekv_percentile(n_tokens_values, 0.95)
```

这些 profile 只在 LPE 可画像对象上创建：

- `h1/run_h1_vllm0110_real.py` 设置 `EDGEKV_H1_STATS_DIR`，并通过 `SamplingParams.extra_args["edgekv_h1"]` 传入 `object_id/reuse_key/object_type/n_tokens/p_reuse/c_recomp_ms/score` 等元数据。
- `h1/sitecustomize.py` patch vLLM `BlockPool.cache_full_blocks()`；只有 `h1_lpe` 且 `_edgekv_should_profile_object()` 返回 true 时，才调用 `_edgekv_profile_from_values()` 创建或更新 `_EDGEKV_GPU_LPE_PROFILES`。
- `h1_lru` 分支只记录 `cached_blocks/admissions` 后直接返回，不创建 profile，所以 `n_tokens_*` 必然为 0。
- `token_block` 默认不画像，除非设置 `H1_LPE_PROFILE_TOKEN_BLOCKS=1`；prefix profile 通常依赖 `H1_PREFIX_REPETITION_PREFIX_LEN` 或显式 request meta。

## stats JSON 数据来源

stats JSON 由 `h1/sitecustomize.py:_edgekv_flush_gpu_cache_stats()` 写出到 `EDGEKV_H1_STATS_DIR/edgekv_gpu_stats_<pid>.json`。每个进程都会按当前本地状态写自己的文件，所以会出现很多只有默认值的 JSON。

主要字段来源如下：

- `lookup_hits`、`lookup_misses`、`lookup_total`、`hit_rate`：patch `BlockPool.get_cached_block()`，prefix cache lookup 命中或未命中时计数。
- `cached_blocks`、`admissions`：patch `BlockPool.cache_full_blocks()`，新 full block 进入 GPU prefix cache 时计数。
- `evictions`、`evict_high_reuse`、`evict_drop`、`low_score_evictions`、`hot_prefix_evictions`、`pinned_eviction_attempts`：patch `BlockPool._maybe_evict_cached_block()`，驱逐时根据 LPE profile 的 `score/p_reuse/object_type/pinned` 分类。
- `touches`：patch `BlockPool.touch()`，缓存块被访问时计数。
- `queue_reorders`、`free_queue_reorder_calls`、`free_queue_reorder_blocks`、`free_queue_reorder_skipped`、`free_queue_reorder_window`、`free_queue_reorder_time_ms`：LPE/LFU 对 vLLM free block queue 做重排时记录。
- `policy_time_ms_total`、`policy_time_us_avg`、`policy_timing_samples`：用 `time.perf_counter_ns()` 统计策略逻辑开销。
- `eviction_decision_time_ms_total`、`eviction_decision_time_us_avg`、`eviction_decision_timing_samples`：统计驱逐决策和 free queue reorder 相关开销。
- `score_min/p50/p95/mean/std`、`p_reuse_min/p50/p95/std`、`c_recomp_ms_min/p50/p95`、`n_tokens_min/p50/p95`：从 `_EDGEKV_GPU_LPE_PROFILES` 聚合；无 profile 时全部为 0。
- `avg_p_reuse`、`avg_score`、`lpe_profile_count`、`cop_object_profile_count`：从 `_EDGEKV_GPU_LPE_PROFILES` 聚合。
- `cop_resident_size_bytes_total`、`cop_resident_size_mb_total`、`cop_resident_size_mb_avg`、`cop_resident_block_count_total`：根据 object 到 resident block 的映射和 vLLM KV cache page size 推导。
- `kv_cache_group_count`、`kv_cache_page_size_bytes_min`、`kv_cache_page_size_bytes_max`：patch KV cache manager 初始化时记录每个 KV cache group 的 page size。
- `object_profiles`：只有设置 `EDGEKV_H1_STATS_INCLUDE_OBJECT_PROFILES=1` 时才写入详细 object profile。
- `pid`、`policy`：当前进程和策略环境。

## aggregate / diagnostics 数据来源

`h1/aggregate_h1_serving_bench.py` 会读取每个 cell 的 `result.json`、`metrics.txt` 和 `stats/edgekv_gpu_stats_*.json`，生成 `aggregate.csv` 和 `lpe_diagnostics.json`。

- `p50_ttft_ms`、`p95_ttft_ms`、`p99_ttft_ms`、`mean_ttft_ms`、`p95_e2el_ms`、`request_throughput`、`output_throughput`、`total_input_tokens`、`total_output_tokens`：来自 serving-bench 的 `result.json`。
- `queue_wait_ms`、`prefill_ms`、`queue_wait_p95_ms`、`prefill_p95_ms`：从 `metrics.txt` 的 Prometheus histogram 解析，匹配 queue/prefill 相关 metric 名称。
- `gpu_prefix_cache_*`、`hit_rate`、`score_*`、`p_reuse_*`、`c_recomp_ms_*`、`policy_time_*`、`cop_*`：从 stats JSON 归并。

当前聚合脚本没有把 `n_tokens_min/n_tokens_p50/n_tokens_p95` 放入 `FLOAT_STAT_KEYS` 和 summary 输出，所以 aggregate CSV/diagnostics 不会保留这些字段。若后续要在 D3 数据中使用 `n_tokens_*`，需要同步扩展 `h1/aggregate_h1_serving_bench.py` 的 stats merge 和 summary 字段。
