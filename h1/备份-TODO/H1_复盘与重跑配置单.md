## 2. 根因：score 公式为什么注定打平 LRU（这段给王祯祥讲透）

你的驱逐分数（§6.2 定义 2）：

```
score(o) = p_reuse(o) · c_recomp(o) / size(o)
```

代入你伪代码里的两条定义 `c_recomp = c_re · n`、`size = μ_kv · n`（n = 对象 token 数）：

```
score(o) = p_reuse(o) · (c_re · n) / (μ_kv · n) = p_reuse(o) · (c_re / μ_kv)
                                    ↑ n 被约掉
```

`c_re/μ_kv` 对同一个模型是**常数**。所以 **score 里唯一变化的量就是 p_reuse**，"按 score 驱逐" ≡ "按 p_reuse 驱逐"。

**直觉（为什么 GreedyDual 在这里失效）**：GreedyDual-Size 的卖点是"留下重算贵的大对象"。但在 KV 缓存里，**重算越贵的块（token 越多），占的显存也越多**，两者同比例增长、单位显存收益一样 → size 归一化把它抵消了。举例：

| 块 | token 数 n | c_recomp | size | score（去掉 p_reuse） |
| --- | --- | --- | --- | --- |
| A 长前缀 | 2000 | 2000·c_re | 2000·μ_kv | c_re/μ_kv |
| B 短前缀 | 200 | 200·c_re | 200·μ_kv | c_re/μ_kv |

A、B 的"每 MB 价值"完全相等，score 区分不了它们。

**真机上更彻底**：vLLM 驱逐单元是**定长 16-token block**，所有候选块的 c_recomp 和 size **字面相等**，score 里连 token 数都没了，只剩 p_reuse。而 p_reuse 你又是用 LRU-K/频次估的 → **LPE 本质就是一个带噪声的 LRU/LFU**，赢不了原版还多了开销和误判。

**数据印证**：理论预测"LPE 必夹在 LRU 和 LFU 之间、永不超过 LRU"——三档数据全部符合。命题成立。

**结论**：要让 score 有意义，**只能引入异质性**（让 `c_re/μ_kv` 不再是常数），唯一现实途径 = 混精度量化块（KIVI/H2O）或不同对象类型。这正是合并稿 H4 的事，也是真正的 CCF-A 门槛。**同质 KV 上调参不可能翻盘。**

> ⚠️ 上面用了你伪代码里"c_recomp 线性、按 block 驱逐"两个前提。重跑前请在代码里确认这俩成立（见 D3）；若你实际实现已不是线性/不是 block 粒度，结论要复核。

## 4. 第二步：换到正确区间重跑（B，**DDL 2026-07-01 出图**）

**目标：把"缓存策略质量"逼成真正的瓶颈。** 需要同时满足两个条件，先记住这两个操作定义：

> - **死区**：命中率 ≥ ~0.95。缓存几乎装得下整个工作集，驱逐谁都不影响命中 → 任何策略都一样。**当前 0.72/0.73 就在死区。**
> - **饱和区**：请求到达速度 > 系统吞吐，请求堆在调度队列里，p95 主要由排队构成（判据：`queue_wait/p95 > 50%`）→ 缓存省的那点 prefill 被排队淹没。**当前 n400 在饱和区。**
> - **有效窗口（要打到这里）**：命中率落在 **0.5~0.85**，且 `queue_wait/p95 < 50%`。这时命中率由驱逐决策决定，策略好坏才看得出来。

**两个旋钮分开拧：**

| 旋钮 | 当前 | 重跑设定 | 目的 |
| --- | --- | --- | --- |
| 缓存显存预算（你 budget 那列）| 0.71~0.73 → 97% 命中 | **往下扫**：`budget ∈ {0.30, 0.40, 0.50, 0.60}`；目标是找到让命中率落进 0.5~0.85 的档 | 制造真实驱逐压力 |
| 负载/到达率 | n400 已饱和 | 用 press 档(n≈200)的到达模式，或显式限并发/限速到吞吐拐点**以下**；用 D4 的 `queue_wait/p95 < 50%` 确认没饱和 | 让 TTFT 反映 prefill 而非排队 |
| 控制变量 | — | 同一条 trace、同模型(Qwen2.5-7B)、同后端、同到达序列；每点 `reps ≥ 3` | 跨策略可比 |

> 如果 budget 已经压不下去（再低就 OOM/拒服务），就改用**另一个旋钮**：把工作集做大（更多不重复会话、降低复用率），同样能把命中率拉下来。二选一，能进有效窗口即可。

**CSV 要记的列**（在你现有 schema 上补齐）：
```
point, budget, policy, reps,
p95_ttft_ms, p50_ttft_ms, mean_ttft_ms, hit_rate,
queue_wait_ms, prefill_ms,            # D4 新增，用于判定是否饱和
gpu_prefix_cache_evictions, gpu_prefix_cache_cached_blocks,
free_queue_reorder_time_ms, policy_time_us_avg,   # D1/D2 修好后才有意义

```
另存一个 side 文件：每个 budget 档的 `score` / `p_reuse` 直方图（D3）。

**要出的图/表：**
- 图1（主图）：x = 命中率（或 budget），y = p95 TTFT，四条线 LRU/LFU/LPE/default。**用竖线标出命中率落入 0.5~0.85 的区段。**
- 图2（健全性）：x = budget，y = 命中率，确认哪些 budget 落进了有效窗口。
- 表1：仅在有效窗口内，逐档算 `LPE vs LRU` 和 `LPE vs LFU` 的 p95 降幅。

**判定（重跑的 Go/No-Go）：**
- ✅ **有信号**：在有效窗口内，LPE 比 LRU/LFU 的 p95 **降 ≥10% 且 ≥2 个 budget 档稳定**，同时 `policy_time_us` 占命中收益 <10% → 值得继续调 θ_keep/reserve 往原判据 ≥20% 推。
- ❌ **无信号（大概率）**：有效窗口内 LPE 仍 ≈ 或劣于 LRU → **同质退化被经验坐实**，停止在 LPE 上投人力，走第 6 节。
