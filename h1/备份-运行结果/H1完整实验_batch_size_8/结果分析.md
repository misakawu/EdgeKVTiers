## 2. 理论预期和当前实验前提不一致

论文规划中，LPE 的定位是 TMS 后的 lifecycle 保底策略：当 TMS 已经把对象降到最低精度后仍超预算时，LPE 按 `score(o, q(o))` 决定 admission/TTL/eviction。

理论位置：

- `论文规划/00_预实验提取.md:511-543`
- 核心公式在 `论文规划/00_预实验提取.md:519-538`：

```text
o* <- argmin score(o, q(o))
on_admit(o_new) <- score(o_new, full) > min_score(resident)
```

其中 score 的含义是：

```text
score(o, q) = p_reuse(o) * c_recomp(o, q) / size(o, q)
```

这个公式要优于 LRU/LFU，必须依赖对象之间存在真实异质性，例如：

- 不同 precision/tier 下的 `size(o, q)` 不同；
- 不同对象的 restore/recompute 代价不是简单按 token 数同比例缩放；
- 不同对象有不同 offload/drop 后果。

但当前 H1 完整实验是在同质 full KV 上比较 LPE/LRU/LFU，没有启用 TMS 的 quality/tier 异质性。因此 score 公式中的 `c_recomp / size` 很容易退化为常数。

如果：

```text
c_recomp = c_re * n_tokens
size = mu_kv * n_tokens
```

则：

```text
score = p_reuse * c_re / mu_kv
```

也就是 score 只剩 `p_reuse` 的缩放，无法提供超过 LRU/LFU 的独立排序信息。

## 3. 当前代码实现中的关键问题

### 3.1 c_recomp 是线性 token 模型

代码位置：

- `h1/sitecustomize.py:733-736`

当前实现：

```python
c_re = _edgekv_env_float('EDGEKV_C_RE_MS_PER_TOKEN', 0.12)
n_tokens = max(int(n_tokens), 1)
c_recomp_ms = float(meta.get('c_recomp_ms', 0.0) or 0.0) or (c_re * n_tokens)
```

这符合 `c_recomp = c_re * n_tokens` 的线性模型。D3 诊断也显示 `c_recomp/n_tokens` 接近常数：

- SharedGPT 0.75: mean = 0.118745767, std = 0.000743833
- SharedGPT 0.9: mean = 0.118745767, std = 0.000743833
- HotQA 0.75: mean = 0.117959605, std = 0.001712735
- HotQA 0.9: mean = 0.117981034, std = 0.001695820

因此，在 fallback theoretical size 口径下，score 会天然退化为 `p_reuse` 的缩放。

### 3.2 fallback size 也是线性 token 模型

代码位置：

- `h1/sitecustomize.py:797-815`

当前实现：

```python
size_mb = float(profile.get('size_mb', 0.0) or 0.0)
if size_mb <= 0.0:
    n_tokens = max(float(profile.get('n_tokens', 1) or 1), 1.0)
    size_mb = _edgekv_env_float('EDGEKV_MU_KV_MB_PER_TOKEN', 0.12) * n_tokens
    profile['size_mb'] = size_mb
    profile['size_source'] = 'fallback_theoretical'
...
profile['score'] = (
    float(profile.get('p_reuse', 0.5))
    * float(profile.get('c_recomp_ms', 0.0))
    / max(size_mb, 1e-9)
)
```

当 `size_source = fallback_theoretical` 时，`c_recomp_ms` 和 `size_mb` 都按 `n_tokens` 线性增长，`n_tokens` 被约掉。

D3 profile 中 fallback theoretical 占比很高：

| cell | profiles | fallback_theoretical | vLLM size |
| --- | ---: | ---: | ---: |
| SharedGPT 0.75 | 271 | 258 | 13 |
| SharedGPT 0.9 | 271 | 112 | 159 |
| HotQA 0.75 | 315 | 285 | 30 |
| HotQA 0.9 | 315 | 40 | 275 |

这解释了为什么 LPE 在 tight 档尤其接近“带噪声的 p_reuse 策略”，而不是理论上的单位显存收益策略。

### 3.3 实际驱逐粒度是 vLLM prefix-cache block，不是完整对象

代码位置：

- LPE 排序 key: `h1/sitecustomize.py:572-579`
- block admission 绑定 object profile: `h1/sitecustomize.py:1396-1425`
- D3 校验逻辑: `h1/validate_d3.py:127-137`

当前 LPE 排序：

```python
if policy == 'h1_lpe':
    return (float(getattr(pool, '_edgekv_h1_scores', {}).get(block_id, 0.0)), recency, block_id)
```

这说明最终参与排序的是 block，而不是论文算法中的完整 object。

D3 诊断确认 eviction granularity：

```text
eviction_granularity: values=['vllm_prefix_cache_block']
```

这和论文 6.4 中 `resident.remove(o*)` 的对象级假设不同。vLLM 中被分配/回收的是固定大小 prefix-cache block；在固定 block 粒度下，所有候选 block 的物理 size 基本相同，`c_recomp/size` 的理论优势进一步消失。

### 3.4 对象级 c_recomp 与 resident block size 混用，导致 score 被放大

这是比“退化为 p_reuse”更具体的实现问题。

代码位置：

- 对象 profile 中保存对象级 `n_tokens` 和 `c_recomp_ms`: `h1/sitecustomize.py:743-752`
- resident block 数变化后，按当前 resident bytes 更新对象 size: `h1/sitecustomize.py:897-923`
- score 使用 `对象级 c_recomp_ms / 当前 resident size_mb`: `h1/sitecustomize.py:797-815`

当前逻辑中，`c_recomp_ms` 通常仍代表完整对象/请求级重算代价，但 `size_mb` 一旦获得 vLLM 信息，就变成当前还 resident 的 block 总大小。于是一个长对象如果只剩 1 个 block，score 会变成：

```text
score ~= p_reuse * full_object_recompute_cost / one_block_size
```

这会异常放大残留长对象的 score，使 LPE 过度保护“只剩少量 block 的长对象残片”，而不是保护真正完整可命中的高价值对象。

实际 profile 例子：

| cell | object type | n_tokens | resident blocks | size_mb | p_reuse | score |
| --- | --- | ---: | ---: | ---: | ---: | ---: |
| SharedGPT 0.9 | sharegpt_hierarchical_prefix | 752 | 1 | 0.015625 | 0.97 | 5602.10 |
| SharedGPT 0.9 | sharegpt_hierarchical_prefix | 736 | 1 | 0.015625 | 0.97 | 5475.46 |
| HotQA 0.9 | rag_chunk_set | 640 | 1 | 0.015625 | 0.94 | 4562.53 |
| HotQA 0.9 | rag_chunk_set | 496 | 1 | 0.015625 | 0.94 | 3479.65 |

这类 score 不再是论文公式里的对象级单位显存收益，而是“完整对象重算代价 / 当前残留块大小”。该口径会扰乱 victim 排序，降低真实 hit。

### 3.5 p_reuse 估计本质上仍是 LRU/LFU 风格启发式

代码位置：

- object type prior: `h1/sitecustomize.py:842-856`
- p_reuse 刷新: `h1/sitecustomize.py:866-894`
- prior weight: `h1/sitecustomize.py:885-887`

当前 `p_reuse` 由以下项混合：

```python
p_freq = hit_count / access_count
p_recency = 1.0 / (1.0 + math.log1p(misses))
p_type = _edgekv_object_type_prior(...)
p_reuse = w_freq * p_freq + w_recency * p_recency + w_type * p_type
```

如果 trace 中没有足够强的未来复用信号，`p_reuse` 实际上只是频次、近因和对象类型先验的混合。它和 LRU/LFU 同源，却增加了估计噪声和策略路径开销，因此很难稳定超过 LRU/LFU。

LPE 的核心问题不是“开销单独解释全部 TTFT 差距”，而是：

1. score 没有带来更高 hit；
2. score 排序还可能因为对象/block 口径混用造成误保护；
3. 在收益不成立时，额外策略开销会直接变成负收益。

## 6. 最终判断

当前 LPE 输给 LRU/LFU 的根因是：

1. 理论假设中 LPE 是 TMS/异质 tier 后的保底策略，但当前实验在同质 full KV 上单独评测 LPE；
2. `c_recomp` 和 fallback `size` 都按 token 数线性增长，`score` 的单位显存收益项退化；
3. vLLM 实际驱逐单位是固定 prefix-cache block，和论文里的 object-level LPE 不一致；
4. 当前实现混用了对象级 `c_recomp_ms` 和当前 resident block size，使长对象残片的 score 被异常放大，排序偏离真实 hit 收益；
5. `p_reuse` 估计本身接近 LRU/LFU 风格启发式，无法补回 score 退化；
6. 在 hit 没提升时，LPE 的 policy/reorder 开销变成纯负担。

因此，这组结果与“LPE 应显著优于 LRU/LFU”的预期不符，主要不是调参问题，而是实验前提与 LPE 理论适用条件不匹配，并且当前实现存在 score 口径不一致。

## 7. 后续建议

短期不建议继续在当前同质 KV + block-level 驱逐设置上死磕 LPE 调参。更合理的路线：

1. 如果继续保留 H1：把结论改成“同质 KV 上 score-only LPE 退化，无法稳定优于 LRU/LFU”，作为负结果和动机。
2. 如果要修 LPE 实现：必须统一 score 口径，要么严格 block-level `c_recomp_block/size_block`，要么实现真正 object-level admission/eviction，避免对象级 cost 除以残留 block size。
3. 如果追求论文主结果：转向 TMS/quality 维度，让 `size(o,q)`、`c_restore(o,q)`、`quality_loss(o,q)` 产生真实异质性。这样 LPE 才符合 `score(o,q)` 的理论使用场景。
4. 在后续实验中单独报告：
   - `score_source`
   - `size_source`
   - `resident_block_count`
   - `score / p_reuse`
   - object-level hit 与 block-level hit 的差异

这些指标可以直接判断 score 是否仍在退化或被 resident fragment 口径污染。
