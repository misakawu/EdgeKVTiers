## 2. 理论预期和当前实验前提不一致

论文规划中，LPE 的定位是 TMS 后的 lifecycle 保底策略：当 TMS 已经把对象降到最低精度后仍超预算时，LPE 按 `score(o, q(o))` 决定 admission/TTL/eviction。

理论位置：

- `论文规划/00_预实验提取.md:511-543`
- 核心公式在 `论文规划/00_预实验提取.md:519-538`：

```text
o* <- argmin score(o, q(o))
on_admit(o_new) <- score(o_new, full) > min_score(resident)
```

其中 score 的含义是：

```text
score(o, q) = p_reuse(o) * c_recomp(o, q) / size(o, q)
```

这个公式要优于 LRU/LFU，必须依赖对象之间存在真实异质性，例如：

- 不同 precision/tier 下的 `size(o, q)` 不同；
- 不同对象的 restore/recompute 代价不是简单按 token 数同比例缩放；
- 不同对象有不同 offload/drop 后果。

但当前 H1 完整实验是在同质 full KV 上比较 LPE/LRU/LFU，没有启用 TMS 的 quality/tier 异质性。因此 score 公式中的 `c_recomp / size` 很容易退化为常数。

如果：

```text
c_recomp = c_re * n_tokens
size = mu_kv * n_tokens
```

则：

```text
score = p_reuse * c_re / mu_kv
```

也就是 score 只剩 `p_reuse` 的缩放，无法提供超过 LRU/LFU 的独立排序信息。

## 3. 当前代码实现中的关键问题

### 3.1 c_recomp 是线性 token 模型

代码位置：

- `h1/sitecustomize.py:733-736`

当前实现：

```python
c_re = _edgekv_env_float('EDGEKV_C_RE_MS_PER_TOKEN', 0.12)
n_tokens = max(int(n_tokens), 1)
c_recomp_ms = float(meta.get('c_recomp_ms', 0.0) or 0.0) or (c_re * n_tokens)
```

这符合 `c_recomp = c_re * n_tokens` 的线性模型。D3 诊断也显示 `c_recomp/n_tokens` 接近常数：

- SharedGPT 0.75: mean = 0.118745767, std = 0.000743833
- SharedGPT 0.9: mean = 0.118745767, std = 0.000743833
- HotQA 0.75: mean = 0.117959605, std = 0.001712735
- HotQA 0.9: mean = 0.117981034, std = 0.001695820

因此，在 fallback theoretical size 口径下，score 会天然退化为 `p_reuse` 的缩放。

### 3.2 fallback size 也是线性 token 模型

代码位置：

- `h1/sitecustomize.py:797-815`

当前实现：

```python
size_mb = float(profile.get('size_mb', 0.0) or 0.0)
if size_mb <= 0.0:
    n_tokens = max(float(profile.get('n_tokens', 1) or 1), 1.0)
    size_mb = _edgekv_env_float('EDGEKV_MU_KV_MB_PER_TOKEN', 0.12) * n_tokens
    profile['size_mb'] = size_mb
    profile['size_source'] = 'fallback_theoretical'
...
profile['score'] = (
    float(profile.get('p_reuse', 0.5))
    * float(profile.get('c_recomp_ms', 0.0))
    / max(size_mb, 1e-9)
)
```

当 `size_source = fallback_theoretical` 时，`c_recomp_ms` 和 `size_mb` 都按 `n_tokens` 线性增长，`n_tokens` 被约掉。

D3 profile 中 fallback theoretical 占比很高：

| cell | profiles | fallback_theoretical | vLLM size |
| --- | ---: | ---: | ---: |
| SharedGPT 0.75 | 271 | 258 | 13 |
| SharedGPT 0.9 | 271 | 112 | 159 |
| HotQA 0.75 | 315 | 285 | 30 |
| HotQA 0.9 | 315 | 40 | 275 |

这解释了为什么 LPE 在 tight 档尤其接近“带噪声的 p_reuse 策略”，而不是理论上的单位显存收益策略。

### 3.3 实际驱逐粒度是 vLLM prefix-cache block，不是完整对象

代码位置：

- LPE 排序 key: `h1/sitecustomize.py:572-579`
- block admission 绑定 object profile: `h1/sitecustomize.py:1396-1425`
- D3 校验逻辑: `h1/validate_d3.py:127-137`

当前 LPE 排序：

```python
if policy == 'h1_lpe':
    return (float(getattr(pool, '_edgekv_h1_scores', {}).get(block_id, 0.0)), recency, block_id)
```

这说明最终参与排序的是 block，而不是论文算法中的完整 object。

D3 诊断确认 eviction granularity：

```text
eviction_granularity: values=['vllm_prefix_cache_block']
```

这和论文 6.4 中 `resident.remove(o*)` 的对象级假设不同。vLLM 中被分配/回收的是固定大小 prefix-cache block；在固定 block 粒度下，所有候选 block 的物理 size 基本相同，`c_recomp/size` 的理论优势进一步消失。

### 3.4 对象级 c_recomp 与 resident block size 混用，导致 score 被放大

这是比“退化为 p_reuse”更具体的实现问题。

代码位置：

- 对象 profile 中保存对象级 `n_tokens` 和 `c_recomp_ms`: `h1/sitecustomize.py:743-752`
- resident block 数变化后，按当前 resident bytes 更新对象 size: `h1/sitecustomize.py:897-923`
- score 使用 `对象级 c_recomp_ms / 当前 resident size_mb`: `h1/sitecustomize.py:797-815`

当前逻辑中，`c_recomp_ms` 通常仍代表完整对象/请求级重算代价，但 `size_mb` 一旦获得 vLLM 信息，就变成当前还 resident 的 block 总大小。于是一个长对象如果只剩 1 个 block，score 会变成：

```text
score ~= p_reuse * full_object_recompute_cost / one_block_size
```

这会异常放大残留长对象的 score，使 LPE 过度保护“只剩少量 block 的长对象残片”，而不是保护真正完整可命中的高价值对象。

实际 profile 例子：

| cell | object type | n_tokens | resident blocks | size_mb | p_reuse | score |
| --- | --- | ---: | ---: | ---: | ---: | ---: |
| SharedGPT 0.9 | sharegpt_hierarchical_prefix | 752 | 1 | 0.015625 | 0.97 | 5602.10 |
| SharedGPT 0.9 | sharegpt_hierarchical_prefix | 736 | 1 | 0.015625 | 0.97 | 5475.46 |
| HotQA 0.9 | rag_chunk_set | 640 | 1 | 0.015625 | 0.94 | 4562.53 |
| HotQA 0.9 | rag_chunk_set | 496 | 1 | 0.015625 | 0.94 | 3479.65 |

这类 score 不再是论文公式里的对象级单位显存收益，而是“完整对象重算代价 / 当前残留块大小”。该口径会扰乱 victim 排序，降低真实 hit。

### 3.5 p_reuse 估计本质上仍是 LRU/LFU 风格启发式

代码位置：

- object type prior: `h1/sitecustomize.py:842-856`
- p_reuse 刷新: `h1/sitecustomize.py:866-894`
- prior weight: `h1/sitecustomize.py:885-887`

当前 `p_reuse` 由以下项混合：

```python
p_freq = hit_count / access_count
p_recency = 1.0 / (1.0 + math.log1p(misses))
p_type = _edgekv_object_type_prior(...)
p_reuse = w_freq * p_freq + w_recency * p_recency + w_type * p_type
```

如果 trace 中没有足够强的未来复用信号，`p_reuse` 实际上只是频次、近因和对象类型先验的混合。它和 LRU/LFU 同源，却增加了估计噪声和策略路径开销，因此很难稳定超过 LRU/LFU。

## 4. LPE 的额外开销放大了劣势

这一节只看一个问题：hit rate 差不多时，LPE 为什么还会慢。

先看策略侧的计时。LPE 的 `policy_time_us_avg` 明显高于 LRU；和 LFU 比，差距主要体现在 reorder 这条路径上。

### SharedGPT

| budget | LPE policy_time_us_avg | LRU policy_time_us_avg | LPE reorder time | LFU reorder time |
| --- | ---: | ---: | ---: | ---: |
| 0.75 | 177.109 us | 6.181 us | 3210.223 ms | 398.978 ms |
| 0.825 | 172.507 us | 6.482 us | 1597.814 ms | 289.035 ms |
| 0.9 | 170.129 us | 7.075 us | 992.417 ms | 219.933 ms |

### HotQA

| budget | LPE policy_time_us_avg | LRU policy_time_us_avg | LPE reorder time | LFU reorder time |
| --- | ---: | ---: | ---: | ---: |
| 0.75 | 133.849 us | 5.058 us | 859.305 ms | 111.438 ms |
| 0.825 | 134.165 us | 16.521 us | 450.981 ms | 77.686 ms |
| 0.9 | 112.358 us | 4.979 us | 175.336 ms | 33.071 ms |

HotQA 的 D4 拆分更直观。`queue_wait_ms` 基本贴地，差距主要进了 `prefill_ms`。

| budget | LPE hit | LRU hit | LFU hit | LPE prefill | LRU prefill | LFU prefill | LPE vs LRU prefill |
| --- | ---: | ---: | ---: | ---: | ---: | ---: | ---: |
| 0.75 | 0.531729 | 0.545353 | 0.534358 | 323.089 ms | 261.190 ms | 281.804 ms | +61.899 ms |
| 0.825 | 0.674215 | 0.674272 | 0.674750 | 247.746 ms | 180.570 ms | 182.326 ms | +67.176 ms |
| 0.9 | 0.731271 | 0.732764 | 0.731271 | 201.814 ms | 149.461 ms | 162.807 ms | +52.353 ms |

这里基本可以排除“排队变慢”。LPE 的 hit rate 也没有明显掉队。剩下的解释很直接：LPE 在 vLLM prefill 过程中挂了更多 Python 控制逻辑，这些逻辑被算进 `scheduled -> first_token`，但这批 workload 里它没有换来更高命中。

下面按代码路径拆开看。

1. `get_cached_block(...)` 不只是查缓存，还会更新对象访问状态。

   代码位置：`h1/sitecustomize.py:1524-1564`

   LPE 命中后会找 block 对应的 object profile，更新访问次数和命中次数；到达 `H1_LPE_SCORE_UPDATE_INTERVAL` 后，还会刷新 `p_reuse`、重算 score，并把 block 重新放进 rank heap。

   ```python
   profile = _edgekv_block_profile(self, block_id, int(group_id))
   refreshed = _edgekv_note_profile_access(profile, hit=True)
   _edgekv_record_lpe_monitor(...)
   if refreshed:
       self._edgekv_h1_p_reuse[block_id] = ...
       self._edgekv_h1_scores[block_id] = ...
       self._edgekv_h1_score_update_seq[block_id] = ...
       _edgekv_heap_touch_block(self, block)
   ```

   LRU/LFU 的 lookup 逻辑轻得多。LFU 需要频次信息，但不需要 object profile，也不需要重算 `p_reuse * c_recomp / size` 这套对象分数。

2. `cache_full_blocks(...)` 在 LPE 下会做一轮对象登记。

   代码位置：`h1/sitecustomize.py:1565-1704`

   每个新 full block 进入 prefix cache 后，LPE 要推断 object id，创建或更新 profile，写 block-object 映射，再把 score、freq、recency 和 heap 状态补齐。

   ```python
   object_id, object_type, n_tokens = _edgekv_infer_object_id(...)
   profile = _edgekv_profile_from_values(object_id, object_type, n_tokens, meta)
   _edgekv_set_block_object(self, kv_cache_group_id, block_id, object_id)
   self._edgekv_h1_p_reuse[block_id] = ...
   self._edgekv_h1_score_update_seq[block_id] = ...
   self._edgekv_h1_scores[block_id] = score
   self._edgekv_h1_freq[block_id] = ...
   self._edgekv_h1_recency[block_id] = ...
   _edgekv_heap_touch_block(self, block)
   _edgekv_record_lpe_monitor('admit', ...)
   ```

   这条路径很容易被忽略，因为它不直接显示成一次大的 reorder。但 HotQA 这种请求数多、block 持续进出的场景里，这些小更新会稳定摊进 prefill。

3. `p_reuse` 刷新不是一个简单计数器更新。

   代码位置：`h1/sitecustomize.py:1029-1058`、`h1/policies/lpe.py:32-70`

   现在的 `p_reuse` 由频次、近因和类型先验混合出来：

   ```python
   p_freq = hit_count / access_count
   p_recency = 1.0 / (1.0 + math.log1p(misses))
   p_type = _edgekv_object_type_prior(...)
   p_reuse = w_freq * p_freq + w_recency * p_recency + w_type * p_type
   _edgekv_recompute_profile_score(profile)
   ```

   score 继续按下面的公式写回 profile：

   ```python
   score = p_reuse * c_recomp_ms / logical_size_mb
   ```

   所以一次刷新会改好几个字段，还会改变 heap 里的排序 key。这个成本不大，但它在 lookup 和 admission 的热路径上反复出现。

4. LPE 和 LFU 都维护 heap，但 LPE 的 key 更贵。

   代码位置：`h1/sitecustomize.py:694-711`、`h1/policies/lru.py:5-11`、`h1/policies/lfu.py:5-13`、`h1/policies/lpe.py:73-92`

   三个策略的差别在这里很清楚。

   `h1_lru` 的 `needs_rank_state = False`。当前实现直接吃 vLLM 原生 free queue 的顺序，不在每次访问时额外维护一个 heap。

   `h1_lfu` 的 `needs_rank_state = True`，但它的 rank key 是 block 级的：

   ```python
   return (float(freq), recency, block_id)
   ```

   `h1_lpe` 也是 `needs_rank_state = True`，不过它的 rank key 来自对象 profile：

   ```python
   return (score, rejected_order, obj_key)
   ```

   每次 `_edgekv_heap_touch_block(...)` 都会把一个新版本压进 lazy heap：

   ```python
   pool._edgekv_h1_blocks[block_id] = block
   version = ...
   pool._edgekv_h1_rank_version[block_id] = version
   heapq.heappush(pool._edgekv_h1_rank_heap, (...))
   pool._edgekv_h1_queue_dirty = True
   ```

   LFU 也走这条路，所以不能把 LPE 的开销简单说成“它需要 heap，LRU 不需要”。更准确的说法是：LFU 的 heap key 只读 `freq/recency/block_id`；LPE 要从 block 反查 object profile，再读 score、admission 标记和 object sort key，而这些 profile 又是前面 lookup/admit 路径额外维护出来的。

   HotQA 的 reorder 数字也符合这个判断。0.75 档 LPE 是 `859.305ms`，LFU 是 `111.438ms`；0.9 档 LPE 是 `175.336ms`，LFU 是 `33.071ms`。差距不是 heap 这个数据结构本身，而是 LPE 的排序依据更重。

5. `get_new_blocks(...)` 分配前会重排 free queue。

   代码位置：`h1/sitecustomize.py:1357-1449`、`h1/sitecustomize.py:1706-1711`

   LPE/LFU 在分配新 block 前都会进这段：

   ```python
   _edgekv_reorder_free_queue(self, num_blocks)
   return original_get_new_blocks(self, num_blocks)
   ```

   reorder 会看 free queue 的头部窗口，补 heap 候选，弹出低 rank block；必要时还会 `queue.remove(...)` 后 `append_n(...)`，再 touch 一遍选中的 block。HotQA 0.75 里 LPE 的 reorder 总时间摊到请求上只有约 1ms，单靠它解释不了 50ms 的 prefill 差距。但它说明 LPE 确实比 LRU 多走了一条驱逐控制路径，也比 LFU 在这条路径上更重。

6. admission 现在还是 diagnostic，没减少数据面工作。

   代码位置：`h1/sitecustomize.py:1186-1247`、`h1/sitecustomize.py:1681-1704`

   这点很关键。当前注释写得很直白：

   ```text
   diagnostic：计算接受/拒绝，但永不阻止 vLLM 原生缓存。
   strict：预留给第二版...
   ```

   也就是说，即使 `_edgekv_evaluate_object_admission(...)` 给出 reject，第一版也只是标记 profile；最多在 reject 时重新 touch 这些 block，让它们在同分时更早被驱逐。它不会跳过 `original_cache_full_blocks(...)`。缓存登记、block 分配、后续 eviction 压力仍然存在。

因此，这批结果里 LPE 的处境比较尴尬：控制面确实做了更多事，但 hit rate 没有变好；score 还受对象/block 口径混用影响，可能保护到错误的残留块。收益没出来时，这些额外更新就直接落到 prefill 上。

## 6. 最终判断

当前 LPE 输给 LRU/LFU 的根因是：

1. 理论假设中 LPE 是 TMS/异质 tier 后的保底策略，但当前实验在同质 full KV 上单独评测 LPE；
2. `c_recomp` 和 fallback `size` 都按 token 数线性增长，`score` 的单位显存收益项退化；
3. vLLM 实际驱逐单位是固定 prefix-cache block，和论文里的 object-level LPE 不一致；
4. 当前实现混用了对象级 `c_recomp_ms` 和当前 resident block size，使长对象残片的 score 被异常放大，排序偏离真实 hit 收益；
5. `p_reuse` 估计本身接近 LRU/LFU 风格启发式，无法补回 score 退化；
6. 在 hit 没提升时，LPE 的 policy/reorder 开销变成纯负担。

因此，这组结果与“LPE 应显著优于 LRU/LFU”的预期不符，主要不是调参问题，而是实验前提与 LPE 理论适用条件不匹配，并且当前实现存在 score 口径不一致。

## 7. 后续建议

短期不建议继续在当前同质 KV + block-level 驱逐设置上死磕 LPE 调参。更合理的路线：

1. 如果继续保留 H1：把结论改成“同质 KV 上 score-only LPE 退化，无法稳定优于 LRU/LFU”，作为负结果和动机。
2. 如果要修 LPE 实现：必须统一 score 口径，要么严格 block-level `c_recomp_block/size_block`，要么实现真正 object-level admission/eviction，避免对象级 cost 除以残留 block size。
3. 如果追求论文主结果：转向 TMS/quality 维度，让 `size(o,q)`、`c_restore(o,q)`、`quality_loss(o,q)` 产生真实异质性。这样 LPE 才符合 `score(o,q)` 的理论使用场景。
4. 在后续实验中单独报告：
   - `score_source`
   - `size_source`
   - `resident_block_count`
   - `score / p_reuse`
   - object-level hit 与 block-level hit 的差异

这些指标可以直接判断 score 是否仍在退化或被 resident fragment 口径污染。
