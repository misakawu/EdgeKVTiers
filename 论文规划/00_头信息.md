## 一句话讲清这是什么（先读这段）

> **EdgeKVLife = 在 vLLM(prefix caching) + LMCache(cache layer / offload 机制) 之上、不改引擎内核的一层「策略层中间件」，把 KV / prefix / RAG-chunk 这些运行时缓存当成有生命周期的资产统一治理（准入 / TTL / 驱逐 / offload / 复用时取回还是重算）。核心技术是 restore-vs-recompute 的临界带宽判据（定理 1），让边缘小显存下 p95 TTFT 更低、且 offload 不被 I/O 反噬。**

这篇工作有**两个核心创新，且分主次**（展开见四章）：

| | 创新 | 一句话 |
| --- | --- | --- |
| **主创新（论文身份）** | 问题重构 | 把运行时缓存重构为**显存受限的在线生命周期治理问题**——缓存即资产，留谁/留多久/踢去哪/复用时取回还是重算，统一进一个目标。 |
| **核心技术创新（硬结果）** | 定理 1 | restore-vs-recompute 的**临界带宽判据** BW=mu_kv/c_re，给「取回还是重算」一个可证、可测的显式判据——这是 InfiniGen / LMCache 等卸载工作缺的那一块。 |

> LPE 的 `score` 驱逐、COP 画像、边缘评测协议都是**支撑这两核**的实现与证据，不单独当卖点。每条创新具体写在论文哪一章，见 `04_论文结构框架.md` 的「创新点 → 论文位置」映射表。

## 这篇工作建在什么之上、我们加了什么（回答「在什么基础上增加内容」）

| 复用、**不重写**（已有地基） | 新增、**即贡献**（你要写的） |
| --- | --- |
| vLLM：paged KV + prefix caching（底层推理引擎） | 策略层中间件 **COP → LPE → RRS**，挂在引擎外围 |
| LMCache：cache layer + GPU↔CPU/SSD offload 通道（机制） | **定理 1** 临界带宽判据 + 在线 `score` 生命周期决策 |
| LRU/LFU/Belady：作为对照基线 | 边缘小显存下的**评测协议**（p95 TTFT / 显存峰值 / 恢复延迟 / 重算比例 / 策略开销）+ 可复现 trace |

> 一句话：**机制（怎么存、怎么搬）复用现成的，贡献在策略（留谁、何时取回还是重算）和它的可证判据。** 不从零写 serving engine、不重做 paged kernel。

## 这个题好不好做（可行性一句话判断）

- **难度**：中等偏高（系统题，工程量主要在真机接入与 trace 管线）。
- **怎么降风险**：**先 trace 仿真、再接真实引擎**——不接模型的解析仿真器 2–3 天就能把 COP/LPE/RRS 闭环跑起来，前 2–3 周即可拿到 H1 结论；真机接入（LMCache/vLLM 钩子）是主要风险点，放在仿真验证之后。
- **最坏退路**：若真机接入受阻或收益不稳，退为**纯 trace 仿真的缓存生命周期评测论文（方案 B，见十章）**，仍能成文。

## 你这篇论文要解决什么问题

你这篇论文围绕 **KV cache、prefix cache、RAG chunk cache 等运行时缓存对象** 展开。核心问题是：边缘端长会话、RAG 和多请求混合负载中，哪些 KV/cache 应保留、迁移、恢复、丢弃或重算，才能同时降低 TTFT 和显存压力。 这里的重点不是把一个开源系统跑起来，也不是简单换一个压缩率或阈值，而是要证明这个运行时对象确实带来成本，并提出一种能解释、能消融、能回退的方法。

## 第一阶段你要先做什么

第一阶段先不要急着写复杂算法。你需要先读懂最强基线，跑通至少两个可比较的 baseline，记录完整的系统指标和质量指标，并用小规模预实验判断这个题目是否成立。只有当瓶颈真实存在、基线结果可信、失败样例能解释时，后续方法设计才有意义。

## 本方向的边界

本题做策略层原型，不从零写 serving engine；重点是 admission、TTL、restore-vs-recompute、offload、失效和观测指标。
