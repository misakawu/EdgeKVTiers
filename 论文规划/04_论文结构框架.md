# 三章 — 论文结构框架
---

## 3.1 论文结构

你后续写论文时，不能把章节写成项目报告。每章都要回答一个审稿问题：为什么重要，现有方法缺什么，你的方法怎么做，实验是否证明有效。

| 章节 | 建议篇幅 | 应该写什么 | 常见问题 |
| --- | --- | --- | --- |
| Introduction | 1.3 到 1.5 页 | 问题、缺口、洞察和贡献。 | 写成背景介绍，没有强基线差异。 |
| Background / Motivation | 1 页 | 证明 KV cache、prefix cache、RAG chunk cache 等运行时缓存对象 是真实瓶颈。 | 只讲概念，没有测量图。 |
| Problem Formulation | 1 页 | 符号、预算、质量约束和优化目标。 | 符号和实验指标对不上。 |
| Method | 2.5 到 3 页 | COP 画像、LPE 生命周期策略、RRS 阈值定理（含证明）三模块 + 伪代码。 | 只有框图，没有决策细节。 |
| Evaluation | 3 到 4 页 | 主实验、消融、参数、端侧、失败案例。 | 只展示平均值，不解释失败。 |
| Related Work | 0.8 页 | 按技术路线分类并突出差异。 | 按时间堆论文。 |

## 3.2 创新点 → 论文位置映射（直接回答「创新具体写在哪里」）

下表把四章的两核 + 支撑落到论文具体位置。**写作时按这张表放，不要把支撑性贡献也写成 Introduction 里并列的创新 bullet。**

| 创新（层级）| 写在 Introduction | 写在 Problem（五章）| 写在 Method（六章）| 写在 Evaluation（七章）|
| --- | --- | --- | --- | --- |
| **主创新**：统一生命周期问题重构 | 贡献 bullet 1 + 洞察段 | §5.3 优化目标 + §5.5 与 baseline 的问题差异 | §6.1 整体流程（COP→LPE→RRS 闭环）| E1（瓶颈真实）+ E2（整体有效）|
| **核心技术**：定理 1 临界带宽判据 | 贡献 bullet 2 | §5.4 定义 3 + 定理 1（含证明）| §6.4 RRS | **E4 带宽–动作阈值曲线**（验证 $BW^\*$）|
| 支撑：LPE `score` 驱逐 | 不单列；并入贡献 1 | §5.4 定义 2 `score` | §6.3 LPE | E2 主表 + E3 消融 |
| 支撑：COP 画像 | 不单列 | §5.1 画像符号 | §6.2 COP | E3 去画像消融 |
| 支撑：边缘评测协议 | 一句带过 | §5.1 指标符号 | — | 贯穿 E1–E6，E5 端侧拆解 |

> 一图记忆：**Introduction 只放两条 bullet（主创新、核心技术）**；定理 1 的证明在 Problem、实现在 Method §6.4、实测验证在 Eval E4——三处呼应，审稿人想否也否不掉。

### 3.3.1 两张关键图：详细设计 + GPT Image 2 生图提示词

> 上表前两行（Motivation 测量图、方法总览图）是论文最重要的两张图，这里展开成可执行的生图规格。**用法**：先用下面的提示词在 GPT Image 2 出一版「构图 + 配色草稿」，再用 matplotlib / draw.io / TikZ 出标签精确的可投稿矢量版。
> ⚠️ **图像模型对长文本、公式、精确数值的渲染不可靠**：提示词里只放极短英文标签；公式（$\mu_{kv}$、$BW^\*$ 等）和数值一律留到矢量工具/LaTeX 里补。生成图只当布局与风格参考，不要直接当终稿。

#### 图 1 — Motivation 图（放 Background/Motivation 节）

**作用**：在读者看 Method 之前就让其接受两件事——① **命中复用能省下整段 prefill**（收益巨大）；② **边缘小显存放不下所有缓存**，朴素策略（LRU/LFU/vLLM 默认）在显存收紧时 p95 TTFT 急剧恶化，这个「恶化缺口」就是生命周期管理的机会空间。直接顶住 §3.4 审稿问「缓存治理在边缘真有空间吗」。数值取自一章 §1.1。

**要画成什么样**（左右两栏，白底，扁平矢量，IEEE 双栏图风格）：
- **Panel (a) 命中的价值**：3 根竖柱比较同一个 1536-token 前缀的首 token 代价——`Recompute ≈184ms`（高）、`Restore ≈19ms`（矮）、`Hit ≈0ms`（极矮）；y 轴 `TTFT (ms)`，柱顶标数值，箭注「hit saves whole prefill」。
- **Panel (b) 显存是真瓶颈**：折线图，x 轴 `GPU memory budget`（紧→松），y 轴 `p95 TTFT (ms)`；三条上扬曲线 `LRU / LFU / vLLM-default`（越紧越陡），一条低平虚线 `Ideal/Belady`；二者之间橙色半透明填充并标注 `opportunity`。
- **配色**：基线灰蓝，机会空间暖橙高亮；字体 sans-serif、标签短。

#### 图 2 — 算法框架图（方法总览图，放 Method §6.1）

**作用**：一图讲清 EdgeKVLife 的系统形态与数据流，并在视觉上把**「引擎机制层（灰，复用、不改）」与「策略层（彩，本文贡献）」分开**——直接回应「在什么基础上加内容」与「不就是换个驱逐策略吗」。让读者看出三模块 COP→LPE→RRS 挂在 vLLM+LMCache 外围、不改内核，且**定理 1（临界带宽）是 RRS 的判据、是核心技术创新**。结构与 §6.1 的 mermaid 一致。

**要画成什么样**（分层块图，白底，扁平矢量）：
- **顶部**：箭头「request / session stream」进入。
- **中部**（高亮彩色大圆角框 `EdgeKVLife policy layer — contribution`）内左→右三模块：
  - `COP`：四维画像 `p_reuse / c_restore / c_recomp / risk_exp` + `score`。
  - `LPE`：显存压力下 `admit / TTL / evict {drop, offload, pin}`，按 score。
  - `RRS`：复用时按**定理 1** 选 `restore / recompute / skip`；醒目徽标 `Theorem 1: BW* = μ_kv/c_re`（高亮）。
- **底部**（灰色框）：`Inference engine: vLLM prefix caching + LMCache offload (reused, unmodified)`；与策略层用 `hooks (on_admit/on_evict/on_reuse)` 箭头相连。
- **闭环数据流**：请求 → 命中检查 → 三支 `{hit / restore(RRS) / recompute or new+admit}` → `COP update` → `LPE under memory pressure` → `trace log (§6.5)`。
- **图例**：灰 = 复用机制（不改），彩 = 本文贡献（策略层 + 定理 1）。

## 3.4 每章要顶住的审稿问题（写时自问）

| 章节 | 审稿人会问 | 不合格的反例 |
| --- | --- | --- |
| Introduction | 这不就是给 vLLM 换个驱逐策略吗？ | 只说"生命周期管理"，不给"对象画像 + restore-vs-recompute 判据 + 边缘小显存"的差异 |
| Motivation | 缓存治理在边缘真有空间吗？ | 只讲概念，没有"显存预算 vs 命中率/p95"的测量 |
| Problem | 优化目标是离线最优还是在线？和 Belady 差多少？ | 不给 Belady 上界对照 |
| Method | RRS 的阈值有依据吗？ | 定理 1 只给结论不给证明/实测验证 |
| Evaluation | offload 会不会反被 I/O 拖慢？ | 不扫带宽、不报 recompute ratio |
| Related Work | 和 SGLang/RAGCache/InfiniGen 差在哪？ | 按时间堆论文，不按"分页/复用/驱逐/卸载"分类 |

## 3.5 写作要求

每一章都要先写目的，再写内容。表格负责承载结论，段落负责解释为什么这些结论成立。不要用口号式表达，例如“提升效率”“保证质量”，而要写出提升的是哪个指标、质量由哪个指标约束、失败时如何处理。
