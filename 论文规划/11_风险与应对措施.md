# 十章 — 风险与应对措施
---

## 10.1 风险矩阵

| 风险 | 触发信号 | 处理方式 |
| --- | --- | --- |
| 复用率太低 | trace 命中率低，缓存收益小。 | 换高复用 workload（长会话/共享前缀），或转 trace-driven 评测。 |
| 策略不如 LRU | E2 中 $score$ 驱逐没明显赢 LRU/LFU。 | 检查 $p_{reuse}$ 估计，必要时退回 LRU 并诚实报告差距。 |
| I/O 反噬 | offload 后复用 $c_{restore}>c_{recomp}$，p95 抬升。 | 严格执行定理 1（低 $BW$ 改重算），限制 offload 场景。 |
| **工程黑洞（最大风险）** | 一上来就和真实引擎/CUDA/显存碎片死磕，数周拿不到任何策略结论。 | **铁律：先 trace 仿真、再接真机**。H1/H2 结论在不接模型的解析仿真器（§0.5.0-pre）里出，2–3 天即可；真机接入放到趋势确认之后（D3 起）。 |
| 接入引擎困难 | LMCache/vLLM 钩子不足以挂策略。 | 先用 trace 仿真器 + 少量真实实验组合（方案 B3）；只接 eviction 钩子也能出 E2/E3。 |
| baseline 复现困难 | 代码不可用或指标对不上。 | 实现等价基线并标注差异。 |
| 故事发散 | 同时碰量化、思考压缩等其他对象。 | 回到"KV/缓存生命周期"这一唯一主对象。 |

## 10.2 方向专属风险

EdgeKVLife 的风险来自它的资源对象：KV cache、prefix cache、RAG chunk cache 等运行时缓存对象。如果这个对象在真实 workload 中并不构成主要成本，或者质量风险无法被观测，那么本题主线就会变弱。前两个月不要急着堆方法，要先证明成本、冗余和风险同时存在。

## 10.3 方案 B

| 优先级 | 方案 | 适用条件 |
| --- | --- | --- |
| B1 | 转为 trace-driven cache 生命周期评测论文（纯仿真，保留主创新 + 定理 1）。 | 真机接入受阻，但仿真器上 H1/H2 趋势成立。 |
| B2 | 只做 LMCache 或 vLLM 上的策略层插件（缩小到单引擎）。 | 真机能接但跨引擎工程量过大。 |
| B3 | 用仿真器 + 少量真实实验组合完成第一篇短文（workshop）。 | 时间不够做完整系统会议长文。 |

**明确触发条件**：§0.5 的 **H1 / H2 / H3 三个假设中有两个无法通过**，即收缩进方案 B（不要继续堆方法）；或连续两周无法证明瓶颈存在、连续三轮消融都不稳定，也主动进入方案 B 讨论。方案 B 不是重新开题，而是复用已有文献、baseline 和实验日志，把题目从大而散改成小而实——**注意：即使退到纯仿真（B1），两个核心创新（统一生命周期问题重构 + 定理 1）仍然成立，论文仍有身份。**
