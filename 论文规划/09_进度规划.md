# 八章 — 进度规划
---

## 8.1 投稿目标与节奏

本方向建议按 12 个月完整线推进。目标会议或期刊可以参考：EuroSys/OSDI/NSDI workshop、MLSys workshop、WWW/ICLR workshop；系统完整性足够时再冲系统会议长文。前 3 个月决定题目是否成立，第 6 个月前完成主实验趋势，第 9 个月形成可投稿初稿，第 12 个月完成正式投稿或可替代版本。**前期（M1–M5）核心结论在 trace 仿真 + 服务器上完成；端侧验证（E5, Jetson / 受限显存 GPU）属 M5 下–M6，视硬件情况推进，不阻塞主线。**

| 月份 | 阶段 | 你要完成什么 | 导师检查点 |
| --- | --- | --- | --- |
| M1 | 文献与仿真器 | 精读 vLLM/Preble/LMCache；写不接模型的解析仿真器（§0.5.0-pre）出 H1/H2 趋势；接 vLLM 默认（开 prefix caching）+ ShareGPT trace 回放，标定 $c_{re}/\mu_{kv}/d_{deser}$ 真值。 | 仿真趋势是否支持继续、问题是否讲清。 |
| M2 | 预实验 + Go/No-Go | H1 结论图（显存–p95 四策略）+ H2 对比表（带宽–p95 三策略）；失败案例；接 LMCache/vLLM 钩子（H3）；端侧 + 服务器策略层收益对比；写 Go/No-Go 报告。 | **P1 门槛**：H1/H2/H3 各通过/不通过。 |
| M3 | 最小原型 + E2 启动 | COP/LPE/RRS 正式版固化 + §6.5 trace 闭环可消融；启动 E2 主实验（vs LRU/LFU/vLLM/Belady）。 | 三模块是否可独立消融、p95 收益是否稳定。 |
| M4 | 消融与参数 | E3 去模块消融（去 COP/去 RRS/去 LPE）；E4 扫 $BW$（验证定理 1 阈值）+ 扫 $\theta_{keep}$/TTL/reserve。 | RRS/LPE/COP 是否各有贡献、阈值预测是否准。 |
| M5 | 端侧验证 | E5：Jetson / 受限显存 GPU 接策略层外围，实测 p95 / $M_{peak}$ / $T_{policy}$。 | **P2 门槛**：端侧收益与 $T_{policy}$ 是否可控。 |
| M6 | 闭环巩固 | 扩 workload 稳定结论；E6 失败案例（I/O 反噬 / 误驱逐热点 / 画像失准）→ 适用边界。 | 闭环结论是否稳健、失败可解释。 |
| M7 到 M9 | 论文初稿 | 写 Related Work、Problem（五章含定理 1 证明）、Method、Evaluation → 可投初稿。 | 是否进入投稿准备。 |
| M10 到 M12 | 补实验与投稿 | E6 补充、附录、复现包，投系统 workshop/会议。 | 是否可以正式投稿。 |

## 8.2 双周推进计划（前 3 月细化 + 后期里程碑）

进度按"**双周**"推进——每两周一行、每行都有可检查交付。前 3 个月（D1–D6）是"题目成不成立"的窗口，逐双周对齐零章 §0.5 的 H0–H3 并衔接 E2；M4 之后改用里程碑式粗排，避免表过长。系统题铁律：**先 trace 仿真拿结论，再花真机时间**（§0.5）。

### 8.2.1 关键窗口期（M1–M3）双周表

逐双周对齐 §0.5 的 H0–H3 + 规则版三模块原型 + E2 启动（与 §0.5.4 的 W1–W8 周排期一一对应，系统题闭环最易卡，故"**先仿真、后真机**"，**D1–D2 不碰真实引擎接入**）：

| 双周 | 月 | 对应预实验/实验 | 你要完成什么 | 双周末交付 | 导师检查点 |
| --- | --- | --- | --- | --- | --- |
| **D1** | M1 上 | 第 0 步 + H0 启动 | 装环境锁版本（`vllm`/`datasets`/`lmcache`）`env.lock`；拉 ShareGPT 200 会话转统一回放 JSON；写不接模型解析仿真器 `sim.py`（§0.5.0-pre），实现 LRU/LFU/score 三策略 + RRS（定理 1），三档 $M_{budget}$ 跑 H1 趋势、多档 $BW$ 跑 H2 趋势；接 vLLM 默认（开 prefix caching）+ trace 回放器 + 指标脚本，**标定真值** $c_{re}/\mu_{kv}/d_{deser}$ | 仿真器 + H1/H2 趋势图（示意标量版）+ 回放闭环 + 真标定值 | 仿真趋势是否支持继续（`score` 紧显存档 p95 < `lru/lfu`、低 $BW$ 下 `rrs` 不劣于固定策略）；最强基线（vLLM/LMCache）是否在复现计划内 |
| **D2** | M1 下 | H1 + H2 正式结论 | 用真标定值替换仿真器示意标量，在三档显存预算×四策略（LRU/LFU/vLLM 默认/score）重跑，画「显存预算–p95 TTFT」四条线（H1 结论图）；扫 $BW\in[0.5,1,2,4,8,16]$ GB/s 比 always-restore/always-recompute/RRS，画「带宽–p95」三条线（H2 对比表）；§6.5 trace JSONL 落盘、可复现 | H1 结论图 + H2 带宽-p95 对比表 + trace 样例 | `score`/`rrs` 趋势是否稳定、同保留率下命中率是否最优 |
| **D3** | M2 上 | 失败案例 + H3 启动 | 从 H1/H2 trace 中挖 3–5 个失败案例（I/O 反噬：低 $BW$ 下 always-restore 的 p95 飙升、误驱逐热点：score 底的高频会话被踢后立即复用、画像失准：$p_{reuse}$ 估计与真实偏差 >30% 的对象）；确认 LMCache/vLLM 暴露的 `on_evict/on_admit/on_reuse` 钩子清单，把 COP/LPE/RRS 挂上去（**先只接 eviction**），单独计时 $T_{policy}$ | 失败案例表（对象/决策/后果/修复建议）+ 策略层可接入 + $T_{policy}$ 初值 | 钩子是否够挂策略、$T_{policy}$ 量级是否 ≪ 命中收益 |
| **D4** | M2 下 | H3 + Go/No-Go | 补接 offload/restore 钩子，策略层完整挂载；服务器 + 端侧（受限显存 GPU 或 Jetson）各跑一遍 H1 的 trace，对比接/不接策略层的 p95 TTFT、hit rate、$M_{peak}$、$T_{policy}$；按 §0.5.4 骨架写两页 Go/No-Go 报告（瓶颈图 + 策略对比表 + RRS 表 + 失败案例表 + 判定） | 端侧拆解表 + Go/No-Go 报告 | **P1 门槛**：H1/H2/H3 各通过/不通过；题目 Go/收缩/切方案 B |
| **D5** | M3 上 | 原型固化 | 按 Go/No-Go 结论固化规则版三模块：COP 画像（$p_{reuse}$ 用 LRU-K、冷启动先验、按 prefix/chunk/session 分类建模）、LPE（小顶堆 $O(\log K)$ 驱逐 + offload/drop/pin 分流 + $\theta_{keep}$ 初值 0.5）、RRS（在线估 $BW$ 滑动平均 + 批量恢复摊薄 $d_{deser}$）；§6.5 trace 闭环可逐模块消融（去画像→退 LRU、去 RRS→总重算、去 LPE→vLLM 默认）；扩 trace 规模（ShareGPT 500+ 会话 + RAG chunk 复用）、多 seed 稳定性 | 可消融原型 + 多 seed 稳定性结果 | 原型是否可直接喂进主实验、消融差异是否显著 |
| **D6** | M3 下 | E2 启动 | 启动 E2 主实验：完整方法 vs 全部基线（LRU/LFU/vLLM 默认/Belady 离线上界）跑三档 $M_{budget}$、两种 workload（多会话 + RAG 混合）；锁实验口径（模型/数据/并发/指标采集），确保同 trace 跨策略可复现 | E2 主结果表初版 + 逼近 Belady 的差距 | p95 TTFT 收益是否稳定、表格数据是否可投 |

> 双周偏差超过一个双周（2 周）就要在组会暴露并补救，不要拖到月底才发现 H1 没通过。D1–D4 即原"前两个月周级计划"的双周合并版，与 §0.5 的 H0–H3 一一对应。本题最大风险是真机接入（工程黑洞），D3 前一定先在仿真器拿到 H1/H2 结论；真机若卡住，按十章决策树**先以仿真器结果为主、少量真机校准为辅**。

### 8.2.2 中后期里程碑（M4–M12，粗排）

题目成立后转入"加规模、加 baseline、加消融、写论文"，按月/阶段给关键里程碑即可，不再逐双周展开：

| 阶段 | 月 | 对应实验 | 关键里程碑 / 交付 |
| --- | --- | --- | --- |
| 消融与参数 | M4–M5 上 | E3、E4 | E3 去模块消融（去 COP→退 LRU/LFU、去 RRS→总重算/总 offload、去 LPE→vLLM 默认）→ Full-vs-去模块对比表；E4 扫 $BW$（验证定理 1 阈值曲线 $BW^\*$）+ 扫 $\theta_{keep}$/reserve/TTL → 参数敏感性曲线族 |
| 端侧验证 | M5 下 | E5 | E5：Jetson / 受限显存 GPU 接 LMCache/vLLM 外围，实测 p95 TTFT / $M_{peak}$ / $T_{policy}$；核对服务器→端侧趋势一致性 → 端侧延迟/显存/开销报告。**P2 门槛**：端侧收益与 $T_{policy}$ 是否可控 |
| 仿真闭环巩固 | M6 | E6 | 扩 workload（ShareGPT 1000+ 会话 / 更多 RAG chunk / 高并发压力）稳定闭环结论；E6 失败案例（I/O 反噬、误驱逐热点、画像失准、显存碎片）→ 失败案例表 + 修复建议 + 适用边界 |
| 论文初稿 | M7–M9 | — | Related Work（按"分页/复用/驱逐/卸载"分类 §0.3）、Problem（五章含定理 1 证明）、Method（六章三模块伪代码）、Evaluation 初稿 → 可投初稿；投稿图表（Motivation 图、算法框架图 §3.3.1、主结果表、带宽-动作阈值曲线、消融表、失败案例表）就绪 |
| 补实验 + 投稿 | M10–M12 | — | 附录（符号表、参数全表、trace 格式）、复现包（代码 + trace + 评测脚本 + README）；投系统 workshop/会议（EuroSys/MLSys/NSDI workshop；完整性足够时冲长文）。**P3 门槛**：是否正式投出/转备选 |

## 8.3 阶段门槛

| 门槛 | 通过标准 | 未通过时怎么办 |
| --- | --- | --- |
| P0 文献门槛 | 三篇最强基线有精读卡和复现计划。 | 暂停方法扩写，先补文献。 |
| P1 瓶颈门槛 | E1 证明运行时对象造成真实成本。 | 转为测量题或更换 workload。 |
| P2 方法门槛 | 完整方法优于固定规则，质量损失可解释。 | 删掉弱模块，保留最强部分。 |
| P3 论文门槛 | 主实验、消融和失败案例能闭合。 | 不强行投稿，转技术报告或 workshop。 |
