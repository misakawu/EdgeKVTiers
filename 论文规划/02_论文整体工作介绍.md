# 一章 — 论文整体工作介绍
---

## 1.1 Motivation

边缘端长会话、RAG、多请求混合负载下，**KV cache、prefix cache、RAG chunk cache** 在运行时持续吃显存：命中能省下整段前缀的 prefill（$c_{recomp}=c_{re}\cdot n$），但显存放不下时必须驱逐，而盲目 offload 又可能在复用时被 I/O 拖慢。问题不是"放得下"，而是**这些缓存对象有生命周期、复用概率、恢复成本和过期风险，需要被当作资产来治理**。

现有方法各管一段：vLLM/PagedAttention 管内存页，SGLang/RadixAttention 做 prefix 复用，LMCache 提供 cache layer，但 workload 层"留谁、踢谁、踢去哪、复用时恢复还是重算"仍多用 LRU/LFU。EdgeKVLife 要证明的是：用对象画像驱动的生命周期策略 + restore-vs-recompute 的临界带宽判据，能在显存受限下显著降 p95 TTFT，且不被 I/O 反噬。

**一个具体的数字（示意，需 E1 实测坐实）**：边缘盒上跑多轮客服会话，一个 1536-token 的会话前缀 KV 约 $\mu_{kv}\cdot n\approx184$ MB；命中复用可省整段 prefill（$c_{re}\approx0.12$ ms/token ⇒ ~184 ms）。但显存只有十几 GB，几十个并发会话就放不下，于是必须驱逐。盲目把它 offload 到 SSD，复用时若 $BW=2$ GB/s 则恢复要 ~92 ms+反序列化，可能比重算（~184 ms）快、也可能更慢——这正是 RRS 要用临界带宽 $BW^\*=\mu_{kv}/c_{re}$ 判定的地方。

## 1.3 方法概述

**地基**：方法**建在 vLLM(prefix caching) + LMCache(cache layer / offload 机制) 之上**，不重写 serving engine、不重做 paged kernel；贡献是挂在引擎外围的一层策略中间件。

EdgeKVLife 是 **COP → LPE → RRS** 的策略层闭环：

| 模块 | 缩写 | 一句话职责 | 需要留下的证据 |
| --- | --- | --- | --- |
| Cache Object Profiler | COP | 为 prefix/chunk/session KV 建 $p_{reuse}/c_{restore}/c_{recomp}/risk_{exp}$ 画像 | 画像、$score$、更新耗时 |
| Lifecycle Policy Engine | LPE | 按 $score$ 决定 admission/TTL/eviction/offload/pin | 驱逐对象、去向、显存占用 |
| Restore-vs-Recompute Scheduler | RRS | 复用时按定理 1 选 restore/recompute/skip | $c_{restore}$ vs $c_{recomp}$、$BW$、动作 |

实现遵循"先 trace 仿真跑通策略，再接 LMCache/vLLM 外围做真实实验"。

## 1.4 贡献点整理（两核 + 支撑，分主次）

| 层级 | 贡献点 | 具体 claim | 证明方式 |
| --- | --- | --- | --- |
| **主创新** | 问题重构 | 将边缘 KV/缓存治理形式化为**显存受限的在线生命周期问题**（缓存即资产，策略层）。 | E1 + 第五章 |
| **核心技术** | RRS 定理 | restore-vs-recompute 临界带宽判据 $BW^\*=\mu_{kv}/c_{re}$，避免 I/O 反噬。 | 定理 1 + E4 多带宽验证 |
| 支撑 | LPE 策略 | $score$ 驱逐 + offload 分流在 trace 上逼近 Belady，优于 LRU/LFU。 | E2 主实验 + E3 消融 |
| 支撑 | 实验协议 | 报告 p95 TTFT、显存峰值、恢复延迟、重算比例、$T_{policy}$。 | E2–E6 + E5 端侧拆解 |

> 写 Introduction 时，前两行各写成一句贡献 bullet（主创新 = 问题重构，核心技术 = 临界带宽判据）；后两行作为支撑，不与前两行并列成「四个创新」。各创新写在论文哪一章见 §3 的「创新点 → 论文位置」映射表。

> 对照 §0.3：LPE 的对象级 $score$ 驱逐区别于 vLLM 的页管理与 H2O 的单序列驱逐；RRS 的临界带宽判据是 InfiniGen/CacheBlend 卸载思路所缺的"何时该重算"显式判据；复用层面与 SGLang/RAGCache 互补——它们做复用，你做"复用 + 留存 + 取回"的统一生命周期。

## 1.5 落地应用与部署目标

| 维度 | 落地设定 |
| --- | --- |
| 目标场景 | 边缘多负载推理服务：本地多轮助手 + RAG，跑在受限显存 GPU / 边缘盒，并发会话多、显存紧 |
| 目标用户痛点 | 显存放不下所有会话/证据 KV，LRU/LFU 命中率低导致频繁重算、p95 抖动 |
| 部署形态 | 作为 **策略层中间件**挂在 LMCache/vLLM 外围，提供 admission/TTL/eviction/offload/restore-vs-recompute；不改 serving engine 内核 |
| 与四人组链路衔接 | 你是"内存中枢"：樊丽宁压输入、俞颉压思考会改变 KV 体量与复用模式，你统一治理这些 KV 的去留；端到端 demo 里你撑起显存与命中率 |
