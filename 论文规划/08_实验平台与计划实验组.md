# 七章 — 实验平台与计划实验组
---

## 7.1 实验平台

| 项目 | 计划配置 | 注意事项 |
| --- | --- | --- |
| 模型 | Qwen2.5-7B-Instruct、Llama-3.1-8B-Instruct、DeepSeek-R1-Distill-Qwen-7B | 先选 1 到 2 个跑通，不要一开始追求最大模型。 |
| 数据或负载 | ShareGPT multi-turn traces、LongBench、HotpotQA、RAG chunk reuse traces、自建多会话 workload | 先做 100 到 300 条小样本闭环。 |
| 软件环境 | Python、PyTorch、Transformers、vLLM/SGLang/方向相关框架。 | 所有版本写入实验记录。 |
| 端侧环境 | Jetson、低资源 GPU 或受限显存设置。 | 端侧样本可少，但必须记录真实开销。 |

## 7.2 全局指标

本方向必须同时报告质量指标和系统指标。建议指标为：TTFT、ITL/TPOT、cache hit rate、GPU memory peak、eviction cost、restore latency、recompute ratio、p95 latency。任何只报告 token 数、压缩率或 cache hit rate 的实验都不够支撑论文结论。

## 7.3 计划实验组

| 实验 | 目的 | 对比对象 / 设置 | 关键指标 | 期望产出物 |
| --- | --- | --- | --- | --- |
| E1 预实验 | 证明命中省 prefill、显存是真瓶颈，拟合 $c_{re}$ | ShareGPT/RAG trace 上测命中收益与显存压力 | $c_{re}$、命中收益、显存曲线 | 瓶颈测量图 + Go/No-Go |
| E2 主实验 | 生命周期策略 vs 强基线 | LRU、LFU、vLLM 默认、（Belady 离线上界）| $TTFT_{p95}$、$hit$、$M_{peak}$ | 主结果表 + 逼近 Belady 的差距 |
| E3 消融 | COP/LPE/RRS 各自贡献 | 去画像（退 LRU）、去 RRS（总重算/总 offload）| $TTFT_{p95}$、$ratio_{recomp}$、$T_{restore}$ | Full vs 去模块对比表 |
| E4 参数敏感性 | 阈值/带宽非单点有效（**验证核心技术创新——定理 1**）| 扫 $BW$（验证定理 1 阈值）、$\theta_{keep}$、TTL、缓存容量 | restore 命中率、$TTFT_{p95}$ | $BW$–动作 阈值曲线 |
| E5 端侧验证 | 策略层在端侧仍有收益 | Jetson/受限 GPU + LMCache/vLLM 外围 | 端侧 $TTFT_{p95}$、$M_{peak}$、$T_{policy}$ | 端侧延迟/显存/开销报告 |
| E6 失败案例 | 何时失败 | I/O 反噬、误驱逐热点、画像失准样本 | 失败 trace、错误类型 | 失败案例表 + 修复建议 |

## 7.4 baseline 复现标准

| baseline 类型 | 最低标准 | 不能接受的做法 |
| --- | --- | --- |
| 无优化基线 | 完整输入、完整推理、默认策略。 | 没有统一参照。 |
| 简单规则 | 固定比例、固定阈值或 LRU/LFU。 | 只和弱规则比较后声称有效。 |
| 强基线 | 至少 2 到 3 个 SOTA 方法或等价实现。 | 只引用不复现。 |
| 端侧基线 | 同模型、同数据、同预算下测量。 | 用服务器结果替代端侧结果。 |

> **实验组与预实验的衔接**：E1 是预实验 H0/H1 的延伸（把 trace 回放扩到全负载 + 拟合 $c_{re}$）；E2/E3 是 H1 的延伸（加 Belady 上界与消融）；E4 是 H2 的延伸（多带宽验证定理 1）；E5 是 H3 的延伸。§0.5 跑通后，E1–E5 是"加规模、加 baseline、加上界对照"。

## 7.5 负载与评测脚本准备

| 资产 | 怎么准备 | 验收 |
| --- | --- | --- |
| 多会话 workload | ShareGPT 多轮 + RAG chunk 复用，转成带时间戳的回放 trace | 同 trace 跨策略复用、可复现 |
| trace 回放器 | 按顺序/时间打请求给 vLLM/LMCache，记录命中/驱逐事件 | 事件落 §6.5 trace |
| 代价标定 | 实测 $c_{re}$（重算/token）、$\mu_{kv}$（MB/token）、$d_{deser}$、各档 $BW$ | 进 COP/RRS 的参数表 |
| Belady 上界 | 离线用未来访问序列算最优驱逐，作为理论上界 | 给 LPE 一个"还差多少"的标尺 |
| 系统计时 | p95 TTFT / 显存峰值统一采集；$T_{policy}$ 单独打点 | 后端/并发锁定写入记录 |

## 7.6 落地与应用

| 落地维度 | 怎么验证 | 关键指标 |
| --- | --- | --- |
| 策略层部署形态 | COP/LPE/RRS 挂 LMCache/vLLM 外围，在受限 GPU/Jetson 跑多会话+RAG | 端到端 p95 TTFT、$M_{peak}$、$T_{policy}$ |
| 同显存收益叙事 | 固定显存预算，比较默认 vs 策略层的 p95 TTFT 与命中率 | 同 $M_{budget}$ 下 p95↓、hit↑ |
| 与链路衔接 | 接收樊丽宁压缩后的输入、俞颉减少的思考 KV，验证整体显存峰值下降 | 链路显存峰值、端到端延迟 |
| 适用边界 | 用 E6 说明"何时策略层不划算"（命中率本就高、显存极充裕） | $T_{policy}$ 占比、I/O 反噬率 |
