# H5 — 质量-带宽相变面验证

## 1. 实验定位

~~`H5` 在 `H0` 无 `LMCache` 模拟器中完成，用于验证最优精度 `q*` 是否会随 `BW` 和 `epsilon` 的变化形成可观测相变面。它验证的是理论预测和算法决策边界，不验证真实系统接入。~~ 修改方案：`H5` 仍在 `H0` 无 `LMCache` 模拟器中完成，但内部执行统一使用 `epsilon_abs`，热力图、网格表和后续论文口径统一使用 `epsilon_norm = epsilon_abs / token_ref`，因此相变面解释更新为 `q*` 随 `BW` 和 `epsilon_norm` 的变化形成可观测分区。

## 2. 回答什么 / 卡住后续什么

本实验要回答：

1. ~~`BW` 与 `epsilon` 两个自变量变化时，`q*_offline` 是否出现清晰分区。~~ 修改方案：`BW` 与 `epsilon_norm` 变化时检查 `q*_offline` 是否出现清晰分区，同时为每个网格点记录对应的 `epsilon_abs` 与 `token_ref`。
2. 定理 1 的解析预测 `q*_pred` 是否与模拟器中离线最优 `q*_offline` 一致。
3. 服务器端和边缘端画像下，相变边界是否方向一致但位置不同。

若 `H5` 失败，定理 1 不能作为 first-class result，只能降级为经验 cost model。

## 3. 前置依赖

1. `pre` 已看到初步 `q*` 分区。
2. `H0` 模拟器支持固定对象集、固定精度三元组和离线穷举。
3. `H4` 已确认 quality 维度至少有算法收益。

## 4. 使用环境与资源

1. 平台：`H0` 模拟器，不接入 `LMCache`。
2. 设备画像：服务器端和边缘端两套 `BW/M_budget/d_deser` 配置。
3. 对象集：从统一 trace 中抽取代表性对象，覆盖短 prefix、长 prefix、高频 RAG chunk、长尾 RAG chunk。
4. ~~网格参数：`BW={0.5,1,2,4,8,16,32}` GB/s，`epsilon={0.5,1,2,4,8}`。~~ 修改方案：网格参数保留 `BW={0.5,1,2,4,8,16,32}` GB/s，但质量预算改为 `epsilon_norm` 网格；内部执行时对每个 `epsilon_norm` 使用 `epsilon_abs = epsilon_norm * token_ref` 换算，旧的 `epsilon={0.5,1,2,4,8}` 仅保留为历史口径说明。
5. 比较器：`q*_offline` 来自模拟器穷举，`q*_pred` 来自定理 1 公式。

## 5. 输入与记录字段

1. `device_profile`
2. `BW`
3. ~~`epsilon`~~ 修改方案：字段改为同时记录 `epsilon_abs`、`epsilon_norm` 与 `token_ref`，其中表格和热力图主轴统一使用 `epsilon_norm`。
4. `object_id`
5. `q*_offline`
6. `q*_pred`
7. `Kendall-tau`

## 6. 实验执行步骤

1. 固定对象集合、精度三元组和质量预算定义。
2. ~~在服务器端画像下扫描 35 个 `(BW, epsilon)` 网格点。~~ 修改方案：在服务器端画像下内部扫描 `(BW, epsilon_abs)` 网格点，但输出表与热力图按 `(BW, epsilon_norm)` 对齐；为固定对象集记录唯一的 `token_ref`。
3. 对每个网格点穷举每对象最优精度 `q*_offline`。
4. 用定理 1 计算 `q*_pred`。
5. 计算 Kendall-tau、一致性比例和异常对象列表。
6. 在边缘端画像下重复扫描，比较相变边界是否合理迁移。
7. 绘制服务器端与边缘端两张热力图。

## 7. 产出物

1. ~~`BW-epsilon` 相变热力图。~~ 修改方案：主图改为 `BW-epsilon_norm` 相变热力图，并在图注或附表中给出每一行 `epsilon_norm` 对应的 `epsilon_abs`。
2. `Kendall-tau` 统计表。
3. 服务器端/边缘端相变边界对照。
4. 异常对象分析报告。

## 8. 通过标准

1. `Kendall-tau >= 0.8`。
2. 热力图呈现明显分区，而不是随机散点或无意义单色。
3. 边缘端相变边界相对服务器端向低带宽/紧预算方向移动，且可由代价模型解释。

## 9. 与后续实验衔接

`H5` 通过后，定理 1 可进入项目构筑和正式实验 `E4`；`H3` 负责检查这些决策能否挂到 `LMCache` 真实 cache layer 上。
