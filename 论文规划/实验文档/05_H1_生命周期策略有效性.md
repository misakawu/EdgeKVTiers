# H1 — 生命周期策略（LPE）有效性

## 1. 实验定位

`H1` 在 `H0` 无 `LMCache` 模拟器中完成，用于验证 `LPE` 的生命周期策略是否优于 `LRU/LFU`。`pre` 只给趋势，`H1` 给算法有效性证据。

## 2. 回答什么 / 卡住后续什么

本实验要回答：

1. `score = p_reuse * c_recomp / size` 是否在相同 trace 和相同显存预算下优于 `LRU/LFU`。
2. `p_reuse`、对象大小和重算代价进入 score 后，是否能解释驱逐对象选择。
3. 服务器端和边缘端画像下，`LPE` 收益是否方向一致。

若 `H1` 不成立，`LPE` 不能作为 `TMS` 之后的 lifecycle 保底模块。

## 3. 前置依赖

1. `pre` 已看到 `score` 相对 `LRU/LFU` 的初步趋势。
2. `H0` 模拟器已可运行，并能记录对象画像和驱逐事件。
3. 统一 trace 中保留 `reuse_key`、访问频次、访问间隔和对象长度。

## 4. 使用环境与资源

1. 平台：`H0` 模拟器，服务器端画像和边缘端画像各跑一遍。
2. Cache 层：不接入 `LMCache`。
3. 数据与负载：统一 `ShareGPT + RAG chunk` trace。
4. 对照策略：`LRU`、`LFU`、`score/LPE`。
5. 自变量：`M_budget` 紧/中/松三档，设备画像两档。
6. 因变量：`TTFT_p95`、`hit_rate`、`M_peak`、`eviction_count`。

## 5. 输入与记录字段

1. `policy`
2. `device_profile`
3. `M_budget`
4. `object_id`
5. `p_reuse`
6. `score`
7. `lpe_action`
8. `TTFT_p95`
9. `hit_rate`

## 6. 实验执行步骤

1. 固定同一 trace、同一设备画像、同一随机种子。
2. 对 `LRU/LFU/score` 分别运行三档 `M_budget`。
3. 记录每次驱逐的 `victim_id`、`score`、`p_reuse`、`size_mb`、`c_recomp`。
4. 分别汇总服务器端和边缘端结果。
5. 绘制 `M_budget -> p95 TTFT` 对比曲线。
6. 抽样解释 `score` 为什么保留或驱逐某些对象。

## 6.1 对应伪代码

```python
def score(o):
    return o.p_reuse * (c_re * o.n_tokens) / o.size_mb

def evict_one(resident, policy):
    if policy == "lru":
        return min(resident, key=lambda o: o.last_used)
    if policy == "lfu":
        return min(resident, key=lambda o: o.freq)
    return min(resident, key=score)
```

## 7. 产出物

1. `H1` 显存预算到 `p95 TTFT` 的对比曲线。
2. 三策略对比表。
3. 服务器端/边缘端趋势对照表。
4. 典型驱逐决策解释样本。

## 8. 通过标准

1. 紧显存档下，`LPE(score)` 相比 `LRU/LFU` 有稳定 `p95 TTFT` 收益。
2. `M_peak` 不超过预算。
3. 驱逐决策可由 `p_reuse/c_recomp/size` 字段解释。

## 9. 与后续实验衔接

`H1` 通过后，`LPE` 可作为 `TMS` 降级之后的保底驱逐模块进入 `H3` 接入验证和正式主实验。
