
# EdgeKVTiers 实验方案设计文档

> 本实验方案基于《王祯祥_论文工作规划.md》中 §0.5 预实验设计、§7.3 实验组、§8 进度规划以及各章节描述，整合形成从预实验到项目构筑、主实验、消融、核心技术验证、端侧验证、失败案例的全套实验流程。所有实验均遵循“先仿真、后真机”原则，并以 H4（quality 维度收益 ≥25%）和 H5（相变面可观测）作为 CCF‑A 路线的硬门槛。

## 统一实验环境基准

后续每份独立实验规划书均以 [00_预实验提取.md](/D:/EdgeKVLife/论文规划/00_预实验提取.md:1) 的预实验口径为准，并区分三层环境：

1. `pre`：纯 Python + trace 数据，不接真实模型、不接 `LMCache`，只判断 `H1/H2/H4/H5` 的变量关系趋势。
2. `H0`：服务器端和边缘端均可运行的无 `LMCache` 模拟器，为 `H1/H2/H4/H5` 验证 `COP/TMS/LPE/RRS` 四个算法有效性提供环境。
3. `H3`：接入 `LMCache/vLLM` 或等价 cache layer，验证策略层可部署性，并为主实验提供真实系统环境。
4. 数据与负载：`ShareGPT` 多轮会话、`HotpotQA/RAG chunk reuse trace`、自建多会话 workload；质量相关实验可附 `LongBench` 子集。
5. 平台：服务器端画像和边缘端画像都必须保留；边缘端可用真实边缘 GPU，也可用服务器限显存/限带宽方式模拟。
6. 必记配置：设备画像、显存预算、`epsilon`、`BW`、随机种子、trace 版本、精度三元组、事件 JSONL 字段。

## 1. 预实验阶段（M1–M2，约 8 周）

预实验的目标：**用最小成本验证核心假设，决定是否继续 CCF-A 路线**。预实验不是一条“全部接真实引擎”的流水线，而是先 `pre` 看变量趋势，再用 `H0` 无 `LMCache` 模拟器验证算法有效性，最后用 `H3` 接 `LMCache` 验证可部署性。

### 1.1 pre - 纯 Python Trace 预仿真

**目的**：使用纯 Python 和 trace 数据模拟 `H1/H2/H4/H5`，判断自变量对因变量是否有稳定影响。

**输入**：
- 精度等级 `Q = {full, int8, int4, sparse_k, offload, drop}`
- 示意三元组 `(size_factor, quality_loss_per_token, restore_factor)`
- 系统参数示意值：`mu_kv`、`c_re`、`d_deser`
- 人工构造或 ShareGPT/RAG 转换得到的访问 trace

**输出**：
- `H1` 初步趋势：`M_budget -> p95 TTFT`
- `H2` 初步趋势：`BW -> p95 TTFT`
- ~~`H4` 初步趋势：`(M_budget, epsilon) -> TMS收益`~~ 修改方案：`H4` 初步趋势改为 `(M_budget, epsilon_norm) -> TMS收益`，并在结果表中保留每档 `epsilon_norm` 对应的 `epsilon_abs`。
- ~~`H5` 初步趋势：`(BW, epsilon) -> q*`~~ 修改方案：`H5` 初步趋势改为 `(BW, epsilon_norm) -> q*`，内部仍按 `epsilon_abs` 执行可行性筛选与代价计算。

**通过条件**：`H1/H2/H4/H5` 至少看到可解释趋势，否则不进入 H0。

---

### 1.2 H0 - 无 LMCache 的服务器端/边缘端模拟器

**目的**：在服务器端和边缘端部署不接入 `LMCache` 的模拟器，为 `H1/H2/H4/H5` 测试 `COP/TMS/LPE/RRS` 四个算法有效性提供统一环境。

**环境**：
- 服务器端画像：较大 `M_budget`、较高 `BW`
- 边缘端画像：较小 `M_budget`、较低 `BW`
- Cache 层：模拟器内部维护 `resident/offloaded/dropped`，不接入 `LMCache`
- 负载：统一 `ShareGPT + RAG chunk` trace

**操作步骤**：
1. 将 `pre` 的 trace 与参数网格固化为配置文件
2. 实现无 `LMCache` 的准入、命中、降级、驱逐、offload、restore/recompute 状态机
3. 暴露 `COP.update`、`TMS.on_pressure`、`LPE.evict`、`RRS.on_reuse` 调用点
4. 在服务器端和边缘端画像下跑 smoke test
5. 输出统一 JSONL 和 summary CSV

**通过条件**：同一 trace 在服务器端/边缘端均能复现，关键自变量可控，事件字段完整。

### 1.3 H4 - Quality 维度核心验证

**方法**：

- 对照：`RAGCache-like`、`static-int4`、`LPE-only`、`TMS-tiered`
- ~~因变量：`TTFT_p95`、`QLoss`、`M_peak`、`N_mig`~~ 修改方案：因变量改为 `TTFT_p95`、`QLoss_abs`、`QLoss_norm`、`M_peak`、`N_mig`，并在表格中同时保留 `epsilon_abs` 与 `epsilon_norm`。

~~**通过门槛**：多数 cell 中 `TMS-tiered` 有显著收益，且 `QLoss <= epsilon`。~~ 修改方案：通过门槛改为多数 cell 中 `TMS-tiered` 有显著收益，且 `QLoss_norm <= epsilon_norm`；实现层保留等价校验 `QLoss_abs <= epsilon_abs`。

### 1.4 H5 - 质量-带宽相变面验证

~~**目的**：在 `H0` 环境中验证最优精度 `q*` 是否随 `(BW, epsilon)` 呈现可解释相变面。~~ 修改方案：在 `H0` 环境中验证最优精度 `q*` 是否随 `(BW, epsilon_norm)` 呈现可解释相变面，并在附表记录对应的 `epsilon_abs`。

**方法**：
1. ~~扫描 `BW={0.5,1,2,4,8,16,32}` GB/s 和 `epsilon={0.5,1,2,4,8}`~~ 修改方案：扫描 `BW={0.5,1,2,4,8,16,32}` GB/s 和 `epsilon_norm` 网格；运行前依据固定对象集 `token_ref` 换算 `epsilon_abs`。
2. 对固定对象集穷举 `q*_offline`
3. 用定理 1 计算 `q*_pred`
4. 分别在服务器端和边缘端画像下计算 Kendall-tau 与热力图

**通过门槛**：Kendall-tau >= 0.8，热力图呈现明显分区。

---

### 1.5 H1 - 生命周期策略（LPE）有效性

**目的**：在 `H0` 环境中验证 `score = p_reuse * c_recomp / size` 是否优于 `LRU/LFU`。

**方法**：
- 平台：`H0` 无 `LMCache` 模拟器
- 自变量：`M_budget` 紧/中/松三档、设备画像两档
- 对照：`LRU`、`LFU`、`score/LPE`
- 因变量：`TTFT_p95`、`hit_rate`、`M_peak`

**通过门槛**：紧显存档下 `LPE(score)` 相比 `LRU/LFU` 有稳定收益。

---

### 1.6 H2 - RRS 恢复 vs 重算阈值

**目的**：在 `H0` 环境中验证 `RRS` 是否能在低带宽下避免 I/O 反噬。

**方法**：
- 平台：`H0` 无 `LMCache` 模拟器
- 自变量：`BW={0.5,1,2,4,8,16}` GB/s
- 对照：`always-restore`、`always-recompute`、`RRS`
- 因变量：`TTFT_p95`、`restore_latency`、`recompute_ratio`

**通过门槛**：多数带宽档下 `RRS` 不劣于两种固定策略中的较优者，低带宽下明显优于 `always-restore`。

---

### 1.7 H3 - 接入 LMCache 的可部署性验证

**目的**：接入 `LMCache/vLLM`，验证 `COP/TMS/LPE/RRS` 作为策略层可部署，并为主实验提供真实系统环境。

**前置依赖**：`H1/H2/H4/H5` 已在 `H0` 中验证算法有效性。

**步骤**：
1. 确认 `LMCache` hook：`on_admit/on_evict/on_reuse/on_pressure`
2. 先接 `COP + LPE + RRS`，稳定后接 `TMS` 降级路径
3. 记录 hook 事件、`T_policy`、`c_mig`、`TTFT_p95`、`M_peak`
4. 对比 `H0` 事件语义与 `H3` hook 事件语义
5. 服务器端跑完整 smoke test，边缘端跑缩小 trace

**通过门槛**：hook 链路跑通，`T_policy` p95 < 1 ms，事件日志完整，服务器端和边缘端均可运行。

---

### 1.8 Go/No-Go 决策报告

报告需分层汇总：
- `pre` 趋势图
- `H0` 无 `LMCache` 服务器端/边缘端环境闭环
- `H1/H2/H4/H5` 算法有效性结果
- `H3` 接入 `LMCache` 的可部署性结果
- 至少 3 个失败案例

**决策规则**：
- **Go**：`pre` 趋势明确，`H0` 通过，`H4/H5` 通过，`H1/H2` 方向稳定，且 `H3` 跑通
- **No-Go**：`H4` 收益过低、`H5` 无相变结构，或 `H0` 无法提供可复现实验环境
- **工程灰色区**：`H0` 算法证据成立但 `H3` 接入不稳，先修接口再进入主实验

## 2. 项目构筑阶段（M3 上，约 3 周）

**目标**：将仿真器中验证的策略逻辑固化为可接入真实引擎的模块化代码，并完成定理证明。

### 2.1 四模块正式实现

| 模块 | 实现内容                                                     | 接口                                                    |
| :--- | :----------------------------------------------------------- | ------------------------------------------------------- |
| COP  | 维护对象画像（`p_reuse`, `ℓ(o,q)`, `size`, `c_recomp`, `c_restore`），支持增量更新 | `update(o, access_event)`                               |
| TMS  | 基于 Hedge/MWU 的在线精度选择；降级优先队列；迁移抖动控制（`Δ_min`） | `on_pressure(resident, ε_rem)` → list of `(o, q_new)`   |
| LPE  | score 驱逐、offload/drop 决策、TTL 管理、pin 名单            | `evict_one(resident)` → `victim_id, action`             |
| RRS  | 依据引理 1 选择 restore/recompute，含在线带宽估计            | `on_reuse(o, q, BW_est)` → `{restore, recompute, skip}` |

**代码要求**：
- 所有决策记录到 §6.6 规定的 JSONL trace 格式
- 支持通过配置开关单独禁用任一模块（用于消融）

**产出**：`edgekvtiers/` 包，包含 `cop.py`, `tms.py`, `lpe.py`, `rrs.py`, `trace_logger.py`

## 3. 主实验 E2（M3 下 – M4，约 6 周）

### 3.1 实验设计

**对比对象**：
- 无优化基线：vLLM 默认（full 精度，无驱逐策略）
- 简单规则：LRU、LFU
- 强 cache 基线：RAGCache（PGDSF 等价实现）
- 强 quality 基线：static‑int4 (KIVI)、static‑sparse‑k (H2O)
- 消融基线：LPE‑only（无 TMS）
- 离线上界：Belady（lifecycle 最优）、Pareto‑OPT（quality × lifecycle 最优，ILP 求解）

**自变量**：
- 显存预算 `M_budget`：紧/中/松 3 档（如 6GB / 10GB / 14GB）
- ~~质量预算 `ε`：紧/中/松 3 档（根据 LongBench 任务允许的 perplexity 增量定义）~~ 修改方案：质量预算对外统一表达为 `epsilon_norm` 的紧/中/松三档；每个 workload/cell 再通过 `token_ref` 换算成内部执行使用的 `epsilon_abs`。
- 负载类型：① ShareGPT 多会话（高前缀复用），② RAG 混合（chunk 复用）

**因变量**（每个 cell 重复 5 次，取 mean±std）：
- ~~**主要**：p95 TTFT、累积 quality loss（必须 ≤ ε）~~ 修改方案：主要指标改为 p95 TTFT、`QLoss_abs`、`QLoss_norm`；正式预算门槛统一写为 `QLoss_norm <= epsilon_norm`。
- **次要**：cache hit rate、GPU 显存峰值、restore latency、recompute ratio
- **调度开销**：`N_mig`（跨精度迁移次数）、`T_policy`、`c_mig` 总开销

### 3.2 执行流程

1. 对每种基线，在相同的 trace 回放器上运行 3×3×2 = 18 个 cell
2. 每个 cell 记录完整 trace JSONL（含每个请求的决策）
3. 收集后处理脚本计算所有指标
4. 生成主结果矩阵表（3×3 网格，每个格子有 6 种基线的 p95/qloss）

### 3.3 产出

- 主实验表格（可放入论文 Evaluation 表 2）
- Pareto 前沿图（横轴 ε，纵轴 p95 TTFT，每条基线一条曲线）
- 箱线图展示 TMS 与基线的 p95 分布差异

## 5. 核心技术验证 E4（M4 – M5 上，约 3 周）

### 5.1 定理 1 验证（相变面）

- 使用与 H5 相同的网格扫描（35 个点），但改用真实引擎（Jetson 或服务器）测量实际 `c_restore`, `c_recomp`
- 对每个 `(BW, ε)`，让 TMS 在线运行 100 个请求，记录每对象最终采用的精度 `q*_online`
- 与离线最优 `q*_offline`（通过穷举得到）计算一致性比例
- 绘制热力图并与定理 1 预测对比

**产出**：图 `fig_phase_transition.pdf`，表 `tab_kendall_tau.pdf`

### 5.2 定理 2 验证（regret 上界）

**步骤**：
1. 固定 `K` 类对象（如按长度分组），`|Q| = 4`（full/int8/int4/sparse_k）
2. 生成不同长度 `T` 的请求序列（`T = 100, 500, 1000, 5000, 10000, 50000`），每个序列重复 10 次
3. 运行 TMS（Hedge，η 按理论值设置），记录每步的累积代价 `C_T`
4. 离线计算 Pareto‑OPT 的最优累积代价 `C*_T`（使用 ILP 或动态规划）
5. 计算 regret `R_T = (C_T - C*_T) / D_max`，绘制 log‑log 图（横轴 T，纵轴 R_T）
6. 拟合斜率，若约等于 0.5 则符合 `O(√T)` 上界

**产出**：regret‑T 曲线图，附理论 bound 线（`√(T·K·log|Q|)`）

## 7. 失败案例与适用边界 E6（M6，约 2 周）

**收集来源**：
- 从 E2/E4 的 trace 中筛选出 p95 高、或迁移次数异常、或质量超标的会话
- 人工分析每个案例的决策轨迹

**案例分类**（至少 3 类，每类 2‑3 例）：
- **降级反弹**：同一对象短时间内反复 full↔int4
- **q* 跨边界抖动**：`(BW, ε)` 靠近相变边界时频繁切换精度
- **I/O 反噬**：RRS 判据因带宽估计不准而错误选择 restore
- **画像失准**：`p_reuse` 估计偏低导致高价值对象被提前降级
- ~~**ε 耗尽**：quality loss 累积超预算，后续对象只能强制 drop~~ 修改方案：`ε` 耗尽统一解释为 `QLoss_norm` 超过 `epsilon_norm`，实现上等价于 `QLoss_abs > epsilon_abs`，失败案例必须保留双口径字段。

**修复与建议**：
- 对每类失败，提出缓解方法（如增大 `Δ_min`、使用滑动窗口带宽估计、引入 hysteresis）
- 最终形成论文中的 **Limitation & Future Work** 小节

**产出**：失败案例表（案例 ID、现象、根因、修复措施、是否已修复）
