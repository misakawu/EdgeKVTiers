### epsilon 的理解与计算

1. **内部执行口径**
   - `epsilon_abs`：内部执行时使用的绝对质量预算。
   - `qloss_abs`：内部执行时累计的绝对质量损失。
   - 当前 `pre` 的对象级质量损失定义为：
     - `qloss_abs(obj, q) = n_tokens(obj) * qloss_per_token(q)`
   - 因此当前 resident/offloaded 对象集合的总质量损失仍按绝对值累加。

2. **归一化口径**
   - `token_ref`：当前实验固定对象集或 candidate set 的总 token 数。
   - `epsilon_norm = epsilon_abs / token_ref`
   - `qloss_norm = qloss_abs / token_ref`
   - 这表示“平均每个 token 允许承受多少质量损失”。

3. **等价关系**
   - 内部执行判定：
     - `qloss_abs <= epsilon_abs`
   - 对外报告判定：
     - `qloss_norm <= epsilon_norm`
   - 两者在固定 `token_ref` 下完全等价，只是量纲不同。

### 当前 pre 中 epsilon 的具体计算方式

1. 先固定对象集，计算：
   - `token_ref = sum(obj.n_tokens for obj in objects)`

2. 在 H4/H5 中先配置归一化预算：
   - `H4`：`epsilon_norm ∈ {0.002, 0.004, 0.008}`
   - `H5`：`epsilon_norm ∈ {0.0005, 0.001, 0.002, 0.004, 0.008}`

3. 运行前换算为内部绝对预算：
   - `epsilon_abs = epsilon_norm * token_ref`

4. 运行结束后再把结果中的绝对质量损失换算为归一化结果：
   - `qloss_norm = qloss_abs / token_ref`

5. 以当前 ShareGPT pre 对象集为例：
   - `token_ref ≈ 72733`
   - 因此 H4 三档约等于：
     - `0.002 -> 145.466`
     - `0.004 -> 290.932`
     - `0.008 -> 581.864`

### 当前 pre 中 epsilon 的意义

1. `epsilon_abs`
   - 表示当前对象集合允许承受的“总绝对质量损失”。
   - 它直接决定：
     - 一个对象能否降到更低精度。
     - TMS 是否还能继续做跨精度降级。
     - H5 中某个精度 tier 是否进入可行集合。

2. `epsilon_norm`
   - 表示平均每个 token 允许承受的质量损失。
   - 它的意义不是改策略，而是让不同 trace、不同 workload、不同设备之间的预算强度可以比较。

3. 为什么需要双口径
   - 只用绝对 `epsilon_abs` 时，预算松紧会强依赖 trace 的总 token 规模。
   - 只用归一化 `epsilon_norm` 时，不便于直接复用当前仿真器内部的绝对累加逻辑。
   - 因此采用：
     - **内部执行**：绝对值。
     - **外部汇总/图表/文档**：归一化值。

### 当前 H0 框架已实现功能

1. Trace 回放闭环
   - H0 能从配置加载 trace，将其转换为 `KVObject` 与 `Request`，并调用 `pre实验/sim.py` 的统一仿真器执行。
   - ShareGPT loader 会将多轮 human turn 转换为 prefix-cache 请求，并额外构造 RAG topic chunk 请求，用于模拟 session prefix 与跨会话 topic reuse。

2. 多设备对照
   - 同一份 trace 只加载一次，然后分别在多个设备画像上回放。
   - 当前已覆盖 server 与 edge 两类画像，便于观察相同 workload 在不同内存、带宽、反序列化成本下的行为差异。

3. 双口径质量预算
   - 配置侧使用 `epsilon_norm`。
   - 运行前根据 `token_ref` 换算：`epsilon_abs = epsilon_norm * token_ref`。
   - 输出同时记录 `epsilon_abs`、`epsilon_norm`、`qloss_current_abs`、`qloss_peak_abs`、`qloss_current_norm`、`qloss_peak_norm`。
   - 因此 H0 与 2026-06-08 后统一的 epsilon/qloss 双口径保持一致。

4. 事件级日志
   - `events.jsonl` 为每个请求补充 `event_index`、`device_profile`、`event`、`object_type`、`n_tokens`、`M_budget`、`BW`、`epsilon_abs`、`epsilon_norm`、`lpe_action`、`tms_action`、`qloss_total_abs`、`qloss_total_norm`、`M_peak`。
   - `event` 字段将底层 hit/miss 与 RRS 行为整理为 `miss`、`resident_hit`、`offload_restore`、`offload_recompute`。

5. 指标级 summary
   - `summary.csv` 记录每个设备画像的一行结果。
   - 关键指标包括 `ttft_p50_ms`、`ttft_p95_ms`、`ttft_mean_ms`、`hit_rate`、`resident_hit_rate`、`offload_hit_rate`、`recompute_ratio`、`restore_ratio`、`memory_peak_mb`、`migrations`、`evictions`、`offloads`、`drops`、`epsilon_ok`。

6. 可复现实验快照
   - `config.resolved.json` 保存实际执行时的配置、trace source、设备画像、对象数、请求数、`token_ref` 与输出文件说明。
   - 顶层 `config.resolved.json` 还包含所有设备 run 的 resolved config 与 validation 结果。
