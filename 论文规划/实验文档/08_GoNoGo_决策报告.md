# Go/No-Go 决策报告

## 1. 文档定位

本报告汇总 `pre/H0/H1/H2/H3/H4/H5` 的证据，决定是否进入项目构筑和主实验。

预实验证据分层如下：

1. `pre`：纯 Python + trace，判断 `H1/H2/H4/H5` 的变量关系是否有趋势。
2. `H0`：无 `LMCache` 的服务器端/边缘端模拟器，提供 `H1/H2/H4/H5` 的算法有效性环境。
3. `H1/H2/H4/H5`：在 `H0` 环境中验证四个算法的有效性。
4. `H3`：接入 `LMCache`，验证可部署性，并为主实验环境放行。

## 2. 报告必须回答的问题

1. `pre` 是否证明关键自变量对因变量有可观测趋势。
2. `H0` 是否提供了服务器端和边缘端均可复现、且不接入 `LMCache` 的模拟器。
3. `H1` 的生命周期策略、`H2` 的恢复/重算阈值、`H4` 的 quality 维度和 `H5` 的相变面是否成立。
4. `H3` 是否证明策略层可接入 `LMCache`，并能支撑主实验。
5. 是否需要切换到收缩方案。

## 2.1 报告所需证据来源

1. `pre` 趋势图：`H1/H2/H4/H5` 的纯 Python 扫描结果。
2. `H0` 环境报告：服务器端和边缘端模拟器配置、trace 版本、JSONL 字段、baseline summary。
3. `H1/H2/H4/H5` 结果：均来自 `H0` 模拟器，不混入 `LMCache` 结果。
4. `H3` 结果：来自 `LMCache/vLLM` 接入 smoke test、hook 日志、`T_policy/c_mig` 拆解和语义一致性检查。
5. 环境记录：设备画像、显存预算、带宽档位、`epsilon`、随机种子、trace 版本、精度三元组。

## 3. 报告结构

1. `pre` 趋势总览：说明哪些变量关系值得继续。
2. `H0` 环境闭环：服务器端/边缘端无 `LMCache` 模拟器是否可复现。
3. `H1/H2/H4/H5` 通过结论：算法有效性证据。
4. `H3` 通过结论：`LMCache` 可部署性证据。
5. 失败案例：至少 3 个来自 `H0` 或 `H3` 的典型问题。
6. 最终决策：`Go`、灰色区复测、或 `No-Go`。

## 4. 决策规则

1. `Go`：`pre` 趋势明确，`H0` 环境通过，`H4/H5` 通过，`H1/H2` 至少方向稳定，且 `H3` 跑通 `LMCache` 接入。
2. `No-Go`：`H4` 收益过低，或 `H5` 无相变结构，或 `H0` 无法提供可复现实验环境。
3. 工程灰色区：`H1/H2/H4/H5` 在 `H0` 中成立，但 `H3` 接入不稳。此时可先进入项目构筑修接口，不进入完整主实验。
4. 算法灰色区：`H3` 可接入，但 `H4/H5` 证据不足。此时不应扩大系统实验，应回到 `pre/H0` 调整问题定义。

## 5. 产出物

1. 两页 Go/No-Go 报告。
2. `pre` 趋势图集合。
3. `H0` 环境复现清单。
4. `H1/H2/H4/H5` 算法有效性表。
5. `H3` 可部署性表。
6. 下一阶段执行建议。

## 6. 与后续实验衔接

`Go` 后进入项目构筑：固化四模块、完善定理证明、扩展 `LMCache + EdgeKVTiers` 原型。若 `H0` 通过但 `H3` 不通过，先修接入接口；若 `H4/H5` 不通过，优先收缩论文主张。
