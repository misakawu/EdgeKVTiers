# 项目构筑 — 接入真实引擎

## 1. 构筑定位

本构筑项对应 [实验方案.md](/D:/EdgeKVLife/论文规划/实验方案.md:215) 的 `2.3`，承接主规划中“先仿真、后真机”的落地阶段。

## 2. 目标

把四模块稳定挂到 `LMCache/vLLM + KIVI/H2O` 的外围，形成可重复运行的 EdgeKVTiers 原型。

## 3. 实现范围

1. `LMCache` 钩子接入
2. `KIVI/H2O` 量化和稀疏化调用
3. offload 存储路径
4. 快速启动与配置脚本

## 3.1 接入所需环境与资源

1. 推理引擎：`vLLM` 作为默认 serving 引擎，开启 prefix caching；保留 `SGLang/RadixAttention` 作为可选后续对照，不作为第一轮接入阻塞项。
2. Cache layer：`LMCache` 负责跨请求 KV 复用、cache/offload 通道和接入钩子；需确认 `on_admit/on_evict/on_reuse/on_pressure` 或等价接口。
3. 模型：服务器侧默认 `Qwen2.5-7B-Instruct`，端侧可使用其 4-bit 版本；接口应避免写死模型结构，后续可切到 `Llama-3.1-8B-Instruct`。
4. 数据与负载：`H0` 统一 `ShareGPT + RAG chunk` trace，最小 smoke test 可用 20-50 会话，正式接入验证使用 100-200 会话。
5. 量化/稀疏化机制：`KIVI/KVQuant` 用于 `int8/int4`，`H2O/SnapKV` 用于 `sparse_k`；若无法 in-place，允许拷贝式实现但必须把 `c_mig` 写入日志。
6. 硬件与存储：服务器 GPU + CPU 内存/SSD offload 路径；记录带宽、序列化/反序列化开销、显存限制方式。
7. 输出配置：提供最小配置文件，包含模型路径、trace 路径、`M_budget`、`epsilon`、`BW_est`、模块开关和日志路径。

## 4. 实现步骤

1. 绑定 `on_admit / on_evict / on_reuse`
2. 接通 `kivi_quantize_inplace` 和 `h2o_sparsify`
3. 打通 restore/offload 数据通路
4. 验证每个动作都有可追踪日志
